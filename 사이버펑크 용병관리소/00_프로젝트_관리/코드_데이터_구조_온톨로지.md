# 코드 데이터 구조 온톨로지

본 문서는 `mvp-web` 내 핵심 데이터의 스키마(`types.ts`), 초기 데이터(`seed.ts`), 그리고 코어 로직(`domain/*.ts`)에서 사용되는 변수들의 매핑과 연관 관계를 간결하게 정리한 사전입니다.

## 1. 지역/등급(Tier) 체계 불일치 (해결 완료 - 2026-06-16)
가장 심각한 버그(생존율 0%)를 유발한 원인이자 전반적인 데이터 파편화의 핵심 지점이었으나 수정 완료됨.

- `Tier` (미션 데이터 및 시스템 전반) = `"lower" | "mid" | "upper"`
- `LocationTier` = 제거됨 (Tier 타입으로 완전 단일화)
- **개선 내용:** 기존의 한글/영문 이원화 매칭 오류를 차단하고, 모든 판정 코어에서 영문 `Tier`를 기준으로 로직이 동작하도록 통일.

## 2. 상태(Status) 및 장비(Gear) 변수 체계
- **용병 상태 (`mercStatuses` in `GameState`)**
  - 저장 형태: `string[]` (상태 ID의 배열)
  - 사용처: 피로도, 부상, **임시 장비 파손 상태** 추적.
  - 주의점: 기존 하드코딩 매직 스트링들을 `constants.ts`의 공통 상수(`STATUS_KIA`, `STATUS_GEAR_DESTROYED` 등)로 통합 관리하도록 개선 완료.
- **장비/임플란트 내구도 및 소유권 정규화 (해결 완료 - 2026-06-16)**
  - 저장 형태:
    - 내구도 상태: `gearStates`, `implantStates` (형태: `Record<string, GearStateValue>`)
    - 소유 관계 매핑: `gearOwner`, `implantOwner` (형태: `Record<string, string>`, 장비 ID별 소유 용병 ID)
  - 타입: `GearStateValue = "worn" | "damaged" | "overheated" | "repaired" | "lost" | "normal"`
  - **개선 내용:** 기존 껍데기 구조에서 벗어나 용병과 장비의 소유 관계 테이블(Record)을 도입. `replaceDestroyedGear`가 실제 파손된 특정 장비의 식별자를 조회하여 `normal`로 복구하는 정교한 수리 경제 루프를 구현.
- **용병 보유 시스템 태그 (`systemTags` in `Mercenary`)**
  - 저장 형태: `string[]` — canonical `tag_id` (`tagDefinitions.ts`)
  - **범위 (Step 0):** 용병 본인 출신·행동·상태만. 절연·은닉·무기 노출 등 **gear/implant 태그는 `entityTagLinks`의 gear/implant 행**으로 분리.
  - 마스터: `tagDefinitions.ts` + 부착 `entityTagLinks.ts`(entityType: mercenary | gear | implant) + 해석 `tagMissionInterpretations.ts`
  - `storyTags?: string[]` — 이야기 태그 분리 필드 (Step B)
  - 판정: `domain/tags.ts` → 해석 규칙 조회 후 GAME_CONFIG 브리지
  - **런타임 출처:** `domain/mercTagPool.ts` — `resolveMercenaryTagAttributions` / `resolveLoadoutTagAttributions` → `TriggeredTag[]` (`sourceType`, `sourceId`)
- **로컬 AI 프록시 제어 (`aiNarratorEnabled` in `GameState`)**
  - 저장 형태: `boolean`
  - 사용처: 로컬 AI 서버로의 비동기 fetch 호출 가드 및 에러 제어, 꺼져 있을 때 즉시 `FALLBACK` 결과로 자동 유도.
- **경제 및 게임 밸런스 설정 격리 (`config.ts` 및 `GAME_CONFIG`)**
  - `src/data/config.ts` 파일의 `GAME_CONFIG` 객체를 통해 초기 자금, 스테이션 업그레이드 비용, 용병 고용 비용 등의 경제 상수를 동적으로 관리. 테스트 및 프로덕션 코드가 이를 동일하게 동적 참조함.

## 3. 스탯 및 판정 시스템
- `StatKey = "frame" | "cool" | "wire" | "cypher" | "pulse"`
- **미션 런 상태 머신 (Step 0, 2026-06-24):**
  - `domain/missionRunExecutor.ts` — `executeMissionRun`: 5단계 파이프라인 범용 뼈대. `engine.simulateMission`이 위임.
  - `domain/nodeQueue.ts` — `NodeQueue`: 동적 노드 큐 (`injectNext`, `injectBeforeExit`).
  - `domain/missionRunContext.ts` — `MissionRunContext`: 런 임시 메모리 (`seizedGearIds`, `allTriggeredTags` 등).
  - `domain/nodeTagJudgment.ts` — `resolveNodeJudgment`: `adverse`/`neutral` + `node_modifier` 해석.
  - `domain/nodePassChance.ts` — `computeNodePassChance`: 스탯 + 장비 보정 + 태그 delta.
  - `RoutingDecision`: `continue` | `goto_phase`(survival) | `inject_node` 등 — 특정 시나리오 하드코딩 없이 분기.
- **노드 데이터 (`MissionNode` / `QueuedNode`):**
  - `challengeTags: string[]` — 이 관문에서 어떤 축으로 태그를 해석할지 (예: `tag_threat_electric`, `tag_challenge_gear_detection`). **빈 배열 = 태그 보정 0.**
  - `defaultPolarity: "adverse" | "neutral"` — adverse는 디폴트 negative(걸리면 손해), neutral은 delta 0.
  - seed 정적 미션·AI 생성 미션에는 아직 대부분 미부여 → 현 플레이는 neutral 위주.
- **판정 로직 (레거시 참고):**
  - `survival.ts`: 생존 판정 시 하층(`frame`, `pulse`), 중층(`cypher`, `cool`), 상층(`cool`, `cypher`)처럼 2개의 스탯 중 최대값을 하드코딩하여 평가.
  - **태그 보너스 보정 (`getTagBonus`):** 용병의 `systemTags`를 기반으로 작전 구역(lower/mid/upper)별 긍정/부정 자질 보유 시 생존율 ±% (생존 축 전용).

## 4. 정산(Settlement) 및 결과(Result) 체계
- `ResultType = "success" | "partial_success" | "failure" | "early_withdrawal" | "incident"`
- **`ResultReport` 확장 (Step 0):**
  - `nodeResolutions?: NodeResolutionLog[]` — 노드별 outcome, triggeredTags, tagDelta.
  - `triggeredTags?: TriggeredTag[]` — 런 전체 발동 태그 (출처 포함).
  - `runContextSnapshot?: MissionRunContext` — `seizedGearIds` 등 런 종료 스냅샷.
  - `/narrate` 프롬프트 주입 **연결 완료** (Phase 4~5-B): `buildNarratePayload` enrich → `App.tsx` POST → `narrate_prompt.py` 「태그 기여 팩트」·「관문별 팩트」·4B 경량화 스켈레톤.
- **조커 카드 로직 (`missionRunExecutor` / `tags.ts`)**
  - 30% 확률로 `resultType`을 `"partial_success"`로 덮어쓰고 `status_gear_destroyed_joker` 부여. loadout gear/implant 태그 포함 판정.
- **생존 로직 (`survival.ts`)**
  - 조커 카드 부재 시(70%), 생존 굴림. 실패 시 `status_kia` 부여.
- **파견 수명 주기 및 복귀 확인 (해결 완료 - 2026-06-18)**
  - 수명 주기: `activeDispatches`(파견 중) ➡️ `completedDispatches`(작전 완료, 정산 대기) ➡️ 정산 완료(돈, 장비 상태 반영되나 용병은 여전히 바쁨 상태 유지) ➡️ **복귀 확인("다음으로" 버튼 클릭)** 시 `completedDispatches`에서 최종 삭제되며 용병 해제.
  - **개선 내용:** 플레이어가 정산 화면에서 "다음으로"를 클릭하지 않고 이탈하더라도 용병이 바쁨 상태로 고정되고, 다시 관제소에서 "복귀 확인"을 통해 안전하게 종결할 수 있게 하여 UI 세션 상태와 용병 상태의 일관성을 완벽히 보장함.

## 5. AI 파이프라인 데이터 플로우

이 섹션은 `narrator_server.py`, `llama-server`, `batch_generate_missions.py`, 프론트엔드 간의 실제 데이터 흐름을 기술한다.

### 추론 아키텍처 (2-tier, 전면 교체 - 2026-06-24)

추론은 더 이상 `narrator_server.py`가 in-process로 모델을 들고 처리하지 않는다.
C++ `llama-server`(연속 배치 엔진)가 추론을 전담하고, `narrator_server.py`는
**프롬프트 조립 + World State 처리 + 프록시** 역할만 한다.

```text
[App.tsx :5173]  ─┐
                  ├─→ narrator_server.py :5001  (프롬프트 조립 + world-state + 프록시)
[batch script]   ─┘                  │
                                     └─→ llama-server.exe :8080  (-np 2, 실제 GGUF 추론)
```

- **왜 교체했나:** 과거 Flask 단일 인스턴스 + `threading.Lock` 직렬화 구조에서는 미션 생성과
  보고서 narrate가 동시에 들어오면 한쪽이 다른 쪽을 막아 30초 타임아웃 → FALLBACK이 떴다.
  실시간 미션 생성을 도입하면 이 충돌이 상시 발생한다.
- **해결:** `llama-server`의 `-np 2` 슬롯이 두 요청을 연속 배치로 병렬 처리한다.
  모델 가중치는 1번만 로드(공유)하고 슬롯별 KV 캐시만 추가되므로 VRAM 효율적(가중치 ~3GB + 약간).
- **엔드포인트:** `narrator_server.py`는 raw `POST /completion`으로 프록시한다.
  (`/v1/chat/completions`는 gemma3 chat 템플릿의 thinking 모드가 켜져 `content`가 비므로 미사용.
  우리는 이미 `<start_of_turn>` 형식 프롬프트를 직접 조립하므로 raw completion이 적합.)
- **바이너리 위치:** `tools/llama-server/llama-server.exe` (llama.cpp b9776, CUDA 12.4 빌드, cudart 동봉).
- **upstream URL:** 기본 `http://127.0.0.1:8080`, 환경변수 `LLAMA_SERVER_URL`로 변경 가능.
- **병렬 검증:** 미션 생성(5.7s)+narrate(2.3s) 동시 발사 시 WALL 5.8s(직렬 예상 8.1s).

### 미션 생성 파이프라인

```text
batch_generate_missions.py (Python, 캐스케이딩 가중 선택으로 미션 뼈대 조립)
  → POST http://127.0.0.1:5001/generate-mission
  → narrator_server.py (프롬프트 조립)
  → POST http://127.0.0.1:8080/completion (llama-server 추론)
  → 응답: { "text": "[목표]: ... [개요]: ... [1차분석]: ... [2차분석]: ..." }
  → src/data/generated_missions.json 저장 (MOCK 모드는 .mock.json 별도 저장)
  → seed.ts: import generatedMissionsRaw from "./generated_missions.json"
  → staticMissions + generatedMissions 병합 export
  → App.tsx: availableMissions (weightMissions 정렬 적용)
  → MissionBoard / MissionDetail 렌더링
```

### 결과 보고서 내러티브 파이프라인

```text
App.tsx (activeDispatches 감시)
  → 미처리 리포트(aiNarrativeKo === undefined) 발견 시
  → 즉시 "GENERATING" 마킹 (중복 요청 방지)
  → buildNarratePayload(report, mission, merc)
      { narrativeMode: catchUpActive ? "fixer_field_log" : "merc_diary", … }
  → POST http://localhost:5001/narrate
      { missionName, mercName, mercId, resultType, summaryLogKo,
        narrativeMode, triggeredTags[], nodeResolutions[] (enriched, intervened 포함), nodeLogs }
  → narrator_server.py → narrate_prompt.py 프롬프트 분기:
      · merc_diary (어사인): build_narrate_system_prompt — 용병 1인칭 일지 + MERC_PERSONAS
      · fixer_field_log (캐치업): build_fixer_field_log_system_prompt — 픽서 1인칭 관찰·감지·지시 기록
  → POST http://127.0.0.1:8080/completion (llama-server 추론, max_tokens 700)
  → 응답: { "narrative": "..." }
  → generatedReports[dispatchId].aiNarrativeKo 갱신
  → ResultReport.tsx: catchUpActive ? 「현장 개입 기록 (관제소 로그)」 : 「현장 보고 (용병 일지)」
  → 실패/타임아웃(90초) 또는 aiNarratorEnabled=false 시
    → fallbackNarrative.ts (모드별: 용병 1인칭 / 픽서 관제 기록)
```

### narrate 프롬프트 계층 구조 (`narrate_prompt.py`)

`narrativeMode`에 따라 **시스템 프롬프트 빌더가 분기**한다. 공통: `build_narrate_user_prompt`(팩트·노드 해상도) + MERC_PERSONAS(merc_diary만).

| `narrativeMode` | 빌더 | 화자·톤 |
| --- | --- | --- |
| `merc_diary` (기본·어사인) | `build_narrate_system_prompt` | 용병 1인칭 현장 일지. 개입 팩트 **미주입** |
| `fixer_field_log` (캐치업) | `build_fixer_field_log_system_prompt` | 픽서 1인칭 관제 기록. **관찰→감지→지시→결과** 골격, `[개입 지시 구간]` 팩트, `FIXER_OBSERVER_STYLE_REFERENCE` |

```text
POST /narrate { narrativeMode, … }
  → narrativeMode === "fixer_field_log"
      ? build_fixer_field_log_system_prompt + user_prompt
      : build_narrate_system_prompt(mercId) + user_prompt
  → <start_of_turn> Gemma3 raw completion → llama-server :8080
```


```text
run_mvp.bat
  → [1] Vite Dev Server 기동 (포트 5173)
  → [2] llama-server 기동 (포트 8080, GGUF 로딩, -np 2)
  → [3] narrator 프록시 기동 (포트 5001)
  → [4] 5001 /health 폴링 (curl -f, 5초 간격, 최대 30회/150초)
       ※ narrator /health 는 upstream(8080) 모델 로딩 완료 시에만 200(ok), 그 전엔 503
  → [5] 브라우저 열기 → batch_generate_missions.py 실행 (생성 완료 시 Vite HMR 자동 리로드)
```

- **health 게이팅:** `narrator_server.py /health`가 준비 전 **HTTP 503**을 반환하므로
  런처/배치가 `curl -f`로 간단히 대기한다. (준비 완료 시에만 200 + `status:"ok"`)
- **127.0.0.1 고정:** `localhost`는 Windows에서 IPv6(::1) 먼저 시도 후 IPv4 폴백까지 ~2.4초
  지연되어 health 체크가 타임아웃 직전까지 늦어지는 문제가 있었다. 모든 내부 호출을
  `127.0.0.1`로 고정(check 0.117s).
- **과거 문제:** 단일 AI 서버 시절 모델 로딩 완료 전 배치 실행 → `/health` 연결 실패 →
  강제 종료. 폴링 루프 + 503 게이팅으로 해결.

## 8. 태그 시스템 데이터 모델 (2026-06-24)

**3계층 데이터 + 4계층 런타임 출처 (Step 0):**

```text
① tagDefinitions.ts     — TagDef 마스터 (속성만, 효과 없음). attachableTo: mercenary|gear|implant
② entityTagLinks.ts     — entityType별 tag_id 부착 (merc / gear / implant 분리)
③ tagMissionInterpretations.ts — 미션 맥락별 reading (긍정|부정|무시|특수)
   statusDefinitions.ts — StatusDef (statuses.csv, TagDef와 별도)

런타임 (Step 0):
④ TriggeredTag — { tagId, reading, sourceType, sourceId, ruleId }
   sourceType: mercenary | gear | implant | status
```

- **타입:** `src/data/types.ts` — `TriggeredTag`, `QueuedNode`, `MissionRunContext`, `RoutingDecision`, `NodeResolutionLog`, `TagJudgmentAxis`(gate_modifier | survival_modifier | joker_card | **node_modifier**), `TagInterpretationContext.challengeTag` 등
- **판정 브리지:** `domain/tags.ts` — 해석 규칙을 `GAME_CONFIG` 수치로 변환
- **노드 태그 판정:** `domain/nodeTagJudgment.ts` — challengeTag × loadout/merc 태그 → adverse/neutral 극성
- **레거시:** `tagRegistry.ts` — `resolveTagId` / `getTagDef` 파사드
- **동기화 (B-0):** `domain/entityTagSync.ts` — merc gameplay 태그 ⊆ `entityTagLinks`(mercenary) ↔ `seed.systemTags`
- **노드 장비 보정 (B-1):** `domain/gearStatBonus.ts` — statBonuses + loadout 합산 → `computeNodePassChance`
- **미션 태그 스텁:** `Mission.missionTags`, `data/missionTagMapping.ts` — danger 키워드→challenge tag_id (가중 생성은 다음 스프린트)

**주의:** 검문·전기 등 특정 기믹은 **고정 노드가 아님**. `challengeTags` + `defaultPolarity`가 생성기가 노드에 붙일 때만 발동.

## 9. 파악된 근본적 문제 리스트 (Refactoring Targets)

1. **Enum String 불일치 (해결 완료):** 영문 `Tier` 하나로 완전 통일하고, UI 표현 단계에서만 한글 룩업 테이블을 거치도록 정렬 완료.
2. **매직 스트링 남용 (해결 완료):** `constants.ts`에 주요 트리거 상태 및 상태 변화 매직 스트링을 상수로 정의하여 교체 완료.
3. **가짜 인벤토리 매핑 (해결 완료):** `gearOwner`/`implantOwner` 관계 데이터를 추가하고 `replaceDestroyedGear` 경제 복구 로직과 연동함으로써, 용병의 전용 장비 상태 파손 및 정비 복구 사이클을 완전히 정규화 완료.
4. **런처 Race Condition (해결 완료 - 2026-06-19):** `run_mvp.bat`이 AI 서버 준비 완료 전에 미션 생성 스크립트를 실행하던 문제. `/health` 폴링 루프 추가로 해결.
5. **AI 추론 직렬화 병목 (해결 완료 - 2026-06-24):** Flask 단일 인스턴스 + `Lock`으로 미션 생성과 보고서 narrate가 동시에 들어오면 한쪽이 막혀 타임아웃 → FALLBACK이 떴다. 추론 엔진을 C++ `llama-server`(`-np 2` 연속 배치)로 분리하고 `narrator_server.py`를 프록시로 전환하여 진짜 병렬 처리 달성. (§5 추론 아키텍처 참고)

## 7. 미션 타입 체계 (2026-06-24 갱신)

- `MissionTypeKey = "잠입" | "전투" | "지원" | "기업" | "비밀" | "교섭" | "추적"` (`types.ts`)
- **`공작` → `교섭`/`추적` 분리:** 어색한 단일 타입 `공작`을 의미가 명확한 `교섭`(협상·포섭)과 `추적`(미행·회수)으로 분리. `MISSION_CHAINS`, `NODE_POOLS`, stat 매핑, TS union, `seed.ts`, `engine.ts`, `state.ts`, 테스트 팩토리 전체 반영 완료.
- 미션 타입은 계층에서 배제되지 않고 **가중치**로만 차등 등장한다(`LAYER_TYPE_BASE_WEIGHTS`). 모든 타입이 모든 계층에서 가능.
- 타깃 팩션은 **하강 규칙**을 따른다: 상위 계층 팩션은 하위 계층 미션의 대상이 될 수 있으나, 하위 계층 팩션이 상위 계층 미션 대상이 될 수는 없다. (`test_target_obeys_descending_rule` 등으로 검증)
