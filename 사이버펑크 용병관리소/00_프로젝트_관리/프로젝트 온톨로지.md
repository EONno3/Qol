# 프로젝트 온톨로지

이 문서는 새 AI가 프로젝트의 개념 관계를 빠르게 파악하기 위한 온톨로지형 색인이다.

원본 설정 문서를 대체하지 않는다. 각 개념이 서로 어떤 관계를 가지는지 보여주는 지도 역할을 한다.

기준 범위는 전체 프로젝트 문서다. 현재 상위 폴더 구조인 `00_프로젝트_관리/`, `01_세계관/`, `02_핵심_시스템/`, `03_태그_사전/`, `04_미션/`, `90_검증_샘플/`, `99_레거시_참고/`를 기준으로 한다.

## 1. 최상위 정의

```text
게임 = 픽서가 용병을 고용하고, 이력서/태그/스탯/장비/장비 속성을 판독해 미션에 파견하는 텍스트 기반 부분 방치형 매니지먼트 게임
```

핵심 플레이 경험:

- 텍스트를 읽고 위험을 추론한다.
- 용병 이력서와 미션 브리핑의 줄글에서 성격, 장점, 위험, 필요한 준비를 스스로 판독한다.
- 수치, 태그, 장비 속성으로 용병/미션 적합성을 판단한다.
- 장비, 임플란트, 스테이션으로 리스크를 통제한다.
- 파견 뒤 결과 보고서의 사건 로그를 읽고 성공/실패 이유와 사건 전개를 다시 판독한다.
- 보상, 상태, 평판, 관계 변화로 다음 판단이 바뀐다.

핵심 주의:

- 이 게임의 중심은 태그 정답 맞추기가 아니라 **글을 읽고 픽서처럼 판단하는 경험**이다.
- 이력서, 미션 카드, 결과 보고서는 모두 플레이어가 해석해야 할 텍스트를 제공해야 한다.
- 분석기와 Phase는 판단을 보조하는 장치이지, 플레이어 대신 정답을 말하는 장치가 아니다.
- Phase가 올라가도 줄글의 판독 가치는 사라지면 안 된다.

## 2. 핵심 루프

```text
캐릭터 생성 (출신지/배경 선택)
→ 스테이션 준비 (인프라 관리 및 용병 고용)
→ 미션 게시판 확인
→ 용병 이력서 판독
→ 스탯/태그/장비/장비 속성/임플란트 비교
→ 미션 매칭 및 용병 파견
→ 미션 노드 판정
→ 결과 보고서
→ 보상/상태/평판/관계 변화 (World State 자동 누적 갱신)
→ 다음 미션과 성장 선택 (자동 저장)
```

## 3. 핵심 엔티티

### 픽서

정의:

- 플레이어.
- 용병을 고용하고 미션에 배치한다.
- 보수 배분, 보험, 장비, 식사, 유흥, 치료, 스테이션 운영을 결정한다.

관계:

- 픽서는 스테이션의 인프라를 관리하고 '용병 고용소'를 통해 용병을 고용/해고한다.
- 픽서는 미션을 수주한다.
- 픽서는 장비와 임플란트를 제공하거나 보험을 건다.
- 픽서는 팩션 평판, 계약 리스크, 용병 만족도를 관리한다.

참조:

- `00_프로젝트_관리/아이디어.md`
- `02_핵심_시스템/사펑용병.html`
- `02_핵심_시스템/스테이션.md`

주의:

- 현재 `02_핵심_시스템/픽서.md`는 이름과 달리 미션 타입과 노드 판정 문서다.

### 용병

정의:

- 미션에 파견되는 유닛.
- 5대 스탯, 장비, 임플란트, 태그, 상태, 이야기 단서를 가진다.

관계:

- 용병은 픽서에게 고용된다.
- 용병은 미션에 파견된다.
- 용병은 스탯을 가진다.
- 용병은 무기와 임플란트를 장착한다.
- 용병은 시스템 태그와 이야기 태그를 가진다.
- **Step 0 (코드):** 절연·은닉·무기 노출 등 **장비/임플란트 태그는 용병 `systemTags`가 아니라 `entityTagLinks`(gear/implant) 소속**. 용병 태그는 출신·행동·상태만.
- 용병은 미션 결과로 신규 상태와 후유증을 얻는다.

참조:

- `02_핵심_시스템/용병.md`
- `03_태그_사전/규칙/태그 분류 규칙.md`
- `03_태그_사전/규칙/이력서 및 미션 카드 압축 규칙.md`

### World State (월드 상태)

정의:

- 픽서 개인의 출신, 스테이션, 자본, 행동 통계, 팩션 평판을 아우르는 전역 게임 상태.

관계:

- 캐릭터 생성 시 출신지에 따라 초기 팩션 호감도와 태그가 세팅된다.
- 매 턴 정산 시 순이익, 수행 미션 타입, 평판 변화가 자동으로 누적 갱신된다.
- `World State`의 출신지와 스테이션 위치는 턴마다 미션 게시판의 가중치 정렬 순서에 영향을 준다.
- 로컬 모델(AI)이 미션 브리핑을 작성할 때 `Layer 2 (현재 World State 스냅샷)`로 주입되어 내러티브에 반영된다.

참조:

- `00_프로젝트_관리/world_state_schema.md`
- `00_프로젝트_관리/ai_ontology_spec.md`

### 스탯

정의:

- 용병의 기본 성능 수치.

종류:

- 프레임: 육체, 하드웨어 수용성, 전투 생존, 중화기.
- 쿨: 침착함, 정신 마모 저항, 협상, 사이버싸이코 억제.
- 와이어: 네트워크, 해킹, 기계 조작, 지원.
- 사이퍼: 은닉, 흔적 제거, 현장 분석, 잠입.
- 펄스: 직관, 돌발 감지, 비정형 위기 대응.

관계:

- 스탯은 기본 관문 돌파 확률의 기반이다.
- 장비와 임플란트는 스탯 요구치나 보정을 가진다.
- 태그는 스탯의 의미를 미션별로 바꾼다.
- 스탯값 자체는 태그가 아니다.
- 스탯 부족/초과는 미션 슬롯 매칭 화면에서 수치/등급/판정으로 해석한다.

참조:

- `02_핵심_시스템/용병.md`
- `02_핵심_시스템/픽서.md`
- `02_핵심_시스템/장비.md`

### 태그

정의:

- 용병, 장비, 미션, 지역, 팩션, 상태, 이야기를 압축하는 단순 정보 단위.
- 태그 자체는 단순해야 한다.
- 복잡한 해석은 미션과 이야기 규칙이 담당한다.

관계:

```text
태그 적용 = 대상 + 관계 + 해석
```

구분:

- 시스템 태그: 미션 판정, 장비, 비용, 상태, 보상에 쓰인다.
- 이야기 태그: 캐릭터 서사, 관계, 비밀, 장기 이벤트에 쓰인다.

원칙:

- 실제 태그 목록을 먼저 대량 생성하지 않는다.
- 공용 태그는 모든 곳에 막 붙는 태그가 아니다.
- 특정 지역/팩션 고유 구조는 전용 태그 규칙의 재료다.
- 태그는 긍정/부정으로 고정하지 않고 미션이 해석한다.
- `#낮은_Frame`, `#높은_Cypher`, `#Wire_경계값`처럼 스탯값을 태그화하지 않는다.
- 장비 노출도는 태그가 아니라 장비 속성/수치 후보인 `Visibility`로 분리한다.
- 상태 태그는 현재 컨디션, 부상, 피로, 각성, 경계심처럼 다음 파견 판단에 영향을 주는 조건이다.
- 상태 태그 변화는 수치 증가가 아니라 태그 신규 부여/전환/완화/해소/소모로 표현한다.
- 상태 태그명은 플레이어가 효과를 직관적으로 추론할 수 있어야 한다.

참조:

### 시스템 태그

정의:

- 게임 판정에 직접 쓰이는 태그.

읽히는 곳:

- 미션 진입 조건.
- 기본 관문 돌파.
- 돌발 노드.
- 조커 카드.
- 장비/임플란트 시너지.
- 장비 속성, 파손, 보험, 유지비.
- 보상, 평판, 수배, 신규 상태.

참조:

- `03_태그_사전/규칙/시스템 태그 규칙.md`

**코드 4계층 출처 (Step 0, `TriggeredTag`):**

```text
mercenary — Mercenary.systemTags + entityTagLinks(mercenary)
gear      — entityTagLinks(gear) + gearOwner loadout
implant   — entityTagLinks(implant) + implantOwner loadout
status    — mercStatuses 등 런타임 상태
```

- AI 서사·결과 로그는 `sourceType` + `sourceId`로 “어떤 장비/용병에서 발동했는지” 추적 가능.
- 태그 효과는 **미션/노드 맥락**(`challengeTag`, tier, missionType)에서 해석 규칙으로만 결정 — 태그 자체에 고정 ±효과 없음.

### 이야기 태그

정의:

- 캐릭터 서사, 관계, 비밀, 장기 이벤트를 여는 태그.

읽히는 곳:

- 이력서 암시.
- 특정 미션 반응.
- NPC/팩션 반응.
- 결과 보고서 단서.
- 후속 이벤트.

주의:

- 이야기 태그는 분석기로 전부 공개하지 않는다.
- Phase 2에서도 일부 핵심 비밀은 숨길 수 있다.

참조:

- `03_태그_사전/규칙/이야기 태그 규칙.md`

### 상태 태그

정의:

- 용병의 현재 상태를 나타내는 시스템 태그의 하위 성격.
- 긍정, 부정, 중립, 조건부 상태를 모두 포함한다.
- 상태는 스탯이 아니므로 `피로 +1단계`처럼 수치 증가로 표현하지 않는다.

변화 방식:

- 신규 부여: `없음 → #경계심`
- 악화/상위 전환: `#피로누적 → #심각한_피로누적`
- 완화/하위 전환: `#심각한_피로누적 → #피로누적`
- 해소/소모: `#경계심 → 제거`
- 병합/변질: 기존 상태와 미션 압력이 합쳐져 다른 상태 태그로 바뀜

주의:

- 임플란트 출력 저하, 필터 오염, 장비 소모는 상태 태그가 아니라 장비/임플란트 변화로 우선 처리한다.
- 장비/임플란트 문제로 생기는 숨참, 기침, 떨림 같은 증상은 결과 로그나 증상 설명으로 둔다.
- 결과 보고서에서 새로 발생하거나 변화한 상태 태그는 Phase와 무관하게 플레이어에게 표시한다.

참조:

- `03_태그_사전/상태 태그 후보군 1차.md`

## 4. 미션

정의:

- 플레이어가 용병을 배치하는 과업.
- 단일 판정이 아니라 여러 관문 노드로 진행된다.

미션 타입 (구현 7종, 2026-06-24):

- 전투: 화력, 생존, 프레임, 장비 내구, 보험.
- 잠입: 사이퍼, `Visibility`, 검문, 은닉 장비, 흔적 제거.
- 지원: 와이어, 수리, 의료, 배송, 기술 조작, 환경 대응.
- 기업: 쿨, 사이퍼, `Visibility`, 팩션 평판, 보안, 인증.
- 비밀: 펄스, 쿨, 미확인 위험, 이야기 태그, 돌발 대응.
- 교섭: 협상·포섭·중재 (구 `공작`에서 분리).
- 추적: 미행·회수·수색 (구 `공작`에서 분리).

> 문서 원본(`02_핵심_시스템/픽서.md`)은 5종 기준이다. 코드(`types.ts`)는 `공작`을
> `교섭`/`추적`으로 분리해 현재 7종이다. 미션 타입은 계층에서 배제되지 않고 가중치로만
> 차등 등장하며, 타깃은 하강 규칙(상위→하위 가능, 하위→상위 불가)을 따른다.

미션 판정 흐름:

```text
미션 타입 확인
→ 필수 조건 확인 (B-2 미착수)
→ 권장 조건 확인
→ 주요 스탯/장비 확인
→ 기본 관문 생성 (NodeQueue — 동적 inject 가능)
→ 관문 돌파 (adverse/neutral + challengeTags + node_modifier)
→ 실패/조건 부족/조건 과잉 시 돌발 노드 (RoutingDecision)
→ 조커 카드 개입
→ 생존 판정 (goto_phase: survival)
→ 결과 보고서와 정산 (triggeredTags, nodeResolutions)
```

**코드 구현 (Step 0):** `MissionRunExecutor` + `NodeQueue` + `MissionRunContext` + `RoutingDecision`. 특정 미션(검문·전기 등) 하드코딩 없음 — 노드의 `challengeTags` + `defaultPolarity` + 해석 규칙 데이터만 사용. seed/AI 미션에 challenge 미부여 시 neutral·delta 0.

### 기본 관문

정의:

- 미션의 기본 진행 단계.
- 스탯, 장비, 시스템 태그를 주로 읽는다.

- 탈출.
- 정산.

### 돌발 노드

- 기본 관문을 대체하거나 끼어든다.

종류:

- 부정 돌발: 조건 부족이나 위험 때문에 발생.
- 긍정 돌발: 권장 조건을 초과 충족해 추가 보상이나 빠른 클리어 발생.

### 무기

구분:

- 외장형 무기: 화력이 높고 바리에이션이 많지만 `Visibility`가 높다.
- 내장형 무기: 임플란트 코스트를 쓰지만 은닉성이 높고 플랜 B가 된다.

관계:

- 무기는 스탯 요구치를 가진다.
- 무기는 임플란트와 시너지를 낸다.
- 외장형과 내장형 동시 장착은 단순 합산이 아니라 역할 분배다.
- `Visibility`는 장비 속성/수치 후보다.
- `Visibility`는 검문/잠입/기업/보안 미션에서 위험값으로 해석될 수 있다.
- `Visibility`가 높은 장비가 항상 부정인 것은 아니다.
- `Visibility`는 시각 노출만이 아니라 발포음, 작동음, 열 배출, 전자 신호, 냄새/화학 흔적, 군용/기업 마킹까지 포함하는 발각 가능성 속성이다.

### 임플란트

구분:

- 바이오: 성능은 낮거나 중간이지만 자가 회복, 낮은 수리비, 낮은 코스트.
- 크롬: 성능과 종류가 좋지만 수리비와 코스트가 높다.
- 기업/군용: 최고급 성능, 높은 코스트, 높은 비용, 강한 세계관/팩션 연결.

주요 파츠:

- 시각.
- 청각.
- 후각.
- 구강.
- 손.
- 상완.
- 하완.
- 고관절/대퇴부.
- 하퇴/발.
- 신경계.
- 호흡계.
- 순환계.
- 소화계.

출격 불가와 강제 조건:

- 시각: 맹인 빌드가 아니면 없을 때 출격 불가.
- 하완: 없거나 완전히 파손되면 모든 미션 출격 불가.
- 하퇴/발: 특수 이동 빌드가 아니면 없거나 파손 시 출격 불가.
- 신경계: 기본 칩셋 없거나 파손 시 출격 불가.
- 호흡계: 일반 미션은 가능, 수중/독가스 등 특정 환경에서 강제 조건.

정산 주의:

- 임플란트 기능 저하를 신체 상태 태그로 대체하지 않는다.
- 필터 오염, 출력 저하, 연결구 이상, 도구 소모는 장비/임플란트 변화로 정산한다.
- 임플란트 문제 때문에 드러난 신체 증상은 결과 보고서의 사건 로그나 증상 설명에 분리해 쓴다.

## 7. 정보 해금

정의:

- 플레이어가 얻는 정보량과 직접성을 단계적으로 늘리는 구조.

Phase:

- Phase 0: 출격 필수 조건 및 파견 여부만 파악 가능. (유리/불리/위험 감지 불가)
- Phase 1: 출격 필수 조건 + 매칭된 태그들의 긍정/부정 요소 '개수'만 요약 노출. 구체적 태그 및 성공 확률 미노출.
- Phase 2: 완벽한 데이터 해금. 시스템 요구, 구체적 태그(유리/불리/치명적 위험 등) 전체 목록 노출 및 예상 성공 확률 공개.

스테이션 분석 기능과의 관계:

- Phase는 추상 정보 강도이며, 실제 해금은 스테이션 기능 레벨에 의해 결정된다.
- 미션 카드의 정보 깊이는 미션 분석 기능 레벨을 따른다.
- 용병 이력서의 정보 깊이는 용병 분석 기능 레벨을 따른다.
- 미션 슬롯 매칭과 결과 보고서는 미션 분석 기능 레벨과 용병 분석 기능 레벨을 함께 참조한다.
- 결과 보고서 Phase 0은 두 분석 기능 중 하나라도 Lv.1 미만일 때 표시된다.
- 결과 보고서 Phase 1은 두 분석 기능이 모두 Lv.1 이상이지만, 둘 중 하나라도 Lv.2 미만일 때 표시된다.
- 결과 보고서 Phase 2는 두 분석 기능이 모두 Lv.2 이상일 때 표시된다.

## 13. 정합성 주의

### Phase 용어

- 정보 해금: Phase 0/1/2.
- 미션 판정: `판정 단계 1~4`로 부른다.
- 결과 보고서 Phase: 스테이션의 미션 분석 기능 레벨과 용병 분석 기능 레벨을 교차 판정한 표시 단계다.

### 레거시 문서

- `99_레거시_참고/태그 및 스탯 시스템 구조안.md`
- `99_레거시_참고/기획 회의 사항.md`
- `02_핵심_시스템/용병.md`의 긍정/부정 태그 분류

이 문서들은 초기 구상이다. 최신 태그 철학과 충돌하면 `03_태그_사전/`을 우선한다.

### 샘플 태그

`99_레거시_참고/미션 & 용병 예시.md`는 레거시 더미 예시다.

태그명, 수치, 성공률뿐 아니라 `기본 분석기`, `고급 분석기`, `상위 분석기`, `모든 데이터 완전 노출`, `미션 정답지` 표현도 최신 기준으로 오해하면 안 된다.

최신 기준은 용병 분석 기관과 미션 분석 기관을 분리하고, 미션 슬롯 매칭은 `min(용병 분석 레벨, 미션 분석 레벨)`을 따른다.

### 테스트 샘플 기준

## 14. AI 작업 원칙

- MVP 우선. 새 규칙이나 새 문서를 늘리기 전에 "이게 MVP에 필요한가?"를 먼저 확인하고, 복잡한 규칙은 고도화 후보로 분리한다.
- 실제 태그를 먼저 대량 생성하지 않는다.
- 태그명보다 판단 질문을 먼저 본다.
- 태그명보다 플레이어가 읽고 판단할 줄글을 먼저 본다.
- 시스템 태그와 이야기 태그를 섞지 않는다.
- Phase 0 문구를 직접적으로 쓰지 않는다.
- 미션은 태그를 고정 효과가 아니라 상황별로 해석한다.
- 스탯값을 태그로 만들지 않는다.
- `Visibility`는 장비 속성으로 다루고, 부정 태그로 만들지 않는다.
- 이력서, 미션 카드, 미션 슬롯 매칭의 정보 위치를 섞지 않는다.
- 결과 보고서 로그를 단순 장식이나 접기용 부가 정보로 취급하지 않는다.
- 결과 보고서에서 손실과 추가 베네핏 미획득을 섞지 않는다.
- 임플란트/장비 기능 저하를 신체 상태 태그로 대체하지 않는다.
- 상층/중층/하층 고유 구조는 공용 태그가 아니라 전용 규칙 재료로 본다.
- 온톨로지는 원본 설정을 대체하지 않는다.
- **시스템 주도 사건 생성 및 AI 윤색 한정**: 로컬 모델의 컨텍스트 한계를 고려하여, AI 프롬프트에 방대한 세계관 요약을 주입하지 않는다. 팩션 간의 역학 관계(하층 인프라 갈등, 의뢰인/타겟 매핑 등)와 발생 사건의 인과관계는 파이썬 코드(시스템)가 온톨로지 로직으로 조립하고 통제해야 한다. 게임 내 AI의 유일한 역할은 시스템이 던져준 갈등 키워드와 뼈대를 '픽서 시점의 건조하고 맛깔나는 첩보 보고서 줄글'로 윤색(포장)하는 것뿐이다.

## 15. AI 파이프라인

정의:

- 시스템 주도 사건(뼈대)을 기반으로 플레이어에게 제공할 텍스트(브리핑, 결과 로그)를 작성하는 구조.
- 게임 내 규칙 개입 권한이 없으며, 오직 '윤색'만 담당한다.

프롬프트 3계층 구조:

- **Layer 1 (정적 지식)**: 세계관 기본 정보 (팩션, 어조 등).
- **Layer 2 (현재 World State 스냅샷)**: 픽서의 명성, 스테이션 정보, 분쟁 중인 팩션, 구역 알림 등 턴마다 갱신되는 게임 진행 상태.
- **Layer 3 (미션/결과 변수)**: 미션 타입, 타겟 팩션, 목표, 위험요소 등 구체적 생성 지시.

추론 아키텍처 (2-tier, 2026-06-24):

```text
[프론트엔드/배치] → narrator_server.py :5001 (프롬프트 조립·world-state·프록시)
                                 → llama-server :8080 (-np 2, 실제 GGUF 추론)
```

- 추론은 C++ `llama-server`(연속 배치 엔진)가 전담하고, `narrator_server.py`는 프롬프트 조립과 프록시만 한다.
- `-np 2` 슬롯으로 미션 생성과 보고서 narrate가 **동시(병렬)** 처리된다. (실시간 미션 생성 대비 기반)
- 모델 가중치는 1번만 로드(공유)하고 슬롯별 KV 캐시만 추가되어 VRAM 효율적.
- raw `POST /completion` 사용. (gemma3 chat 템플릿 thinking 모드 회피)
- **Phase 4~5-B (완료):** `ResultReport.triggeredTags`·`nodeResolutions`를 `buildNarratePayload`로 enrich → `/narrate` 프롬프트에 「태그 기여 팩트」·「관문별 팩트」 주입. Phase 5-B에서 Gemma 4B용 경량화(규칙 4개·스켈레톤·max_tokens 700). 상세: `narrate_프롬프트_경량화_워크스루.md`.

관계:

- AI 스크립트(`batch_generate_missions.py`)는 `--state`를 통해 Layer 2 데이터를 주입받아 미션 프롬프트와 윤색을 수행한다.
- 미션 뼈대(장소·타입·타깃·의뢰인·난이도)는 파이썬의 **캐스케이딩 가중 선택**으로 조립되며, 타깃은 **하강 규칙**(상위→하위 가능, 하위→상위 불가)을 따른다.
- 이를 통해 픽서의 이전 선택(팩션 호감도 변화 등)이 다음 미션 브리핑의 내러티브에 자연스럽게 녹아들게 한다.
