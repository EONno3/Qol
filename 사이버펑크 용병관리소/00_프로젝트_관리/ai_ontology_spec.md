# AI 온톨로지 명세 (AI Ontology Spec)

> **목적:** AI(로컬 GGUF 모델)가 내러티브를 생성할 때 주입받는 정보의 구조와 규칙을 정의한다.  
> **기준일:** 2026-06-11  
> **연관 문서:** `world_state_schema.md`, `static_data_spec.md`, `batch_generate_missions.py`

---

## 핵심 원칙

> [!IMPORTANT]
> AI는 **윤색기(Narrator)**다. 시스템이 먼저 구조를 만들고, AI는 그걸 읽고 플레이어가 읽는 텍스트로 변환한다.  
> AI가 게임 규칙을 결정하지 않는다. AI가 스탯 수치나 태그를 직접 언급하지 않는다.

---

## 1. 정보 계층 구조

AI에게 주입되는 정보는 3개 레이어로 구성된다.

```
[Layer 1] 정적 세계관 지식 (항상 주입, 변하지 않음)
  → 도시 구조, 팩션 정의, 용어 체계, 어조 규칙

[Layer 2] 현재 World State 스냅샷 (턴마다 갱신, 생성 직전 주입)
  → 팩션 현재 상태, 구역 상황, 픽서 명성/악명, 최근 이벤트

[Layer 3] 생성 대상 변수 (미션/용병/결과별로 다름)
  → 미션 뼈대 JSON, 용병 스펙 JSON, 결과 판정 데이터
```

## 2. Layer 1 — 정적 세계관 지식

### 2.1 도시 구조 (압축 버전)

```text
[도시 기본 구조]
가로 ~20km × 세로 ~4km 돔 도시. 4대 메가코프 치외법권 운영.
- 하층(Tier 3): ~600만 인구. 슬럼·쓰레기장·인프라. 법과 보호 없음. UNzone/Pipe/Line/Vent 구역.
- 중층(Tier 2): ~350만. 4대 메가코프 영지. 계획도시. K-Sec 치안. 독립 팩션 다수.
- 상층(Tier 1): ~50만. 특권층·CEO 가문. 바이오 스캔·DNA 인증 보안.
```

### 2.2 팩션 압축 정의

```text
[메가코프]
- 바이루이: 의료·식품 독점. 바이오 임플란트. 인체 실험 의혹.
- 마사다: 군수·금융 카르텔. 크롬·무기·보험. 위험을 가격화.
- 키쿠모토: 전뇌망 OS·정보 통제. 해킹·미디어. 내부 AI 권한 분쟁.
- 알-마디낫: 공간·물류·치안 독점. 건축·보안. 돔 설계 결함 은폐.

[하층 인프라 팩션]
- 더 널스(UNzone): 하층 빈민·난민 집단.
- 더 밸브(Pipe): 수압·식수 통제. 화학 폐수 위험.
- 더 퓨즈(Line): 전력 배분 통제. 고압 누전 위험.
- 더 댐퍼(Vent): 공기·기온 통제. 인공 돌풍·협소 공간 위험.

[중층 독립 팩션]
- La Nada: 유흥·BD(브레인댄스) 시장.
- No Claim: 불법 물류.
- Plus One: 의전·암살.
- Rattle: 정보망·해커 집단.
- One More: 임플란트 개조.
- Ten Men: 중재·평판 브로커.
```

### 2.3 어조 규칙 (AI 글쓰기 제약)

```text
[공통 어조 규칙]
1. 스탯 수치(Frame, Cool, Wire 등)를 직접 언급하지 않는다.
   → "Frame이 높아야 한다" 금지
   → "저 격벽을 힘으로 밀어야 한다" 허용
2. 태그명을 직접 언급하지 않는다.
   → "#슬럼_출신이 유리하다" 금지
   → "하층 뒷골목 출신이라면 길을 알 것이다" 허용
3. 감정이 배제된 건조한 어조. 냉소적·하드보일드 톤.
4. 환경과 상황을 묘사해서 플레이어가 스스로 추론하게 만든다.
5. 브리핑: 150~250자, 3~5문장. 결과 로그: 300~500자.
```

## 3. Layer 2 — World State 스냅샷 (동적 주입)

생성 직전 현재 world_state에서 추출해 프롬프트에 삽입.

### 3.1 스냅샷 구조

```json
{
  "current_turn": 14,
  "fixer_summary": {
    "codename": "픽서 코드명",
    "origin": "하층_언존_태생",
    "fame": 35,
    "infamy": 10,
    "expertise_tags": ["#전장_브로커"],
    "station": "중층 | 마인드 팰리스"
  },
  "notable_faction_states": [
    {
      "faction": "더 밸브",
      "status": "더 퓨즈와 분쟁 중 (강도 3/5)",
      "fixer_rep": 20
    },
    {
      "faction": "바이루이",
      "status": "안정",
      "fixer_rep": -40
    }
  ],
  "area_alerts": [
    "하층 Pipe 구역: 폐수 누출 위험, 전투 미션 빈도 급증",
    "중층 상가 지구: 정상 운영"
  ],
  "recent_player_actions": [
    "더 밸브 전투 미션 성공 (3턴 전)",
    "바이루이 연관 용병 파견 실패 (7턴 전)"
  ]
}
```

### 3.2 스냅샷 압축 규칙

- 팩션 상태: **분쟁 중이거나 픽서 호감도 절댓값 30 이상인 팩션**만 포함
- 구역 알림: **현재 미션 생성에 영향을 주는 구역**만 포함
- 최근 행동: **최근 5턴 이내 주요 결정**만 포함
- 전체 토큰 예산: 스냅샷은 **500토큰 이하**로 압축

## 4. Layer 3 — 생성 대상별 변수 및 프롬프트 규칙

### 4.1 미션 브리핑 생성

**입력 (시스템 → AI):**

```json
{
  "mission_type": "잠입",
  "mission_detail": "탈취",
  "client_faction": null,
  "target_faction": "키쿠모토",
  "goal_type": "기밀",
  "location": {
    "layer": "중층",
    "zone": "상가 지구",
    "structure": "보안 게이트",
    "spot": "상가 지구 2번 게이트"
  },
  "danger": "케이섹 무선 식별 핑, 갱단 골목 매복",
  "primary_stat": "Cypher",
  "secondary_stat": "Cool",
  "phase_0_visible": ["mission_type", "location", "rough_reward"],
  "phase_1_visible": ["danger_direction", "stat_tier_hint"],
  "phase_2_visible": ["requirements", "nodes", "exact_reward"]
}
```

**출력 요구사항:**

| Phase | AI가 생성할 내용 | 길이 |
|---|---|---|
| Phase 0 (브리핑) | 환경·상황 묘사. 위험을 암시하되 수치 없음 | 150~250자, 3~5문장 |
| Phase 1 (신호) | 방향성 힌트 문장. "이런 환경이다"까지만 | 50~80자, 1~2문장 |
| Phase 2 (요약) | 요구 조건과 위험의 근거 설명 | 100~150자, 2~3문장 |

**Phase 0 브리핑 Few-shot 예시:**

```
입력: [잠입, 탈취, 의뢰인 없음, 키쿠모토, 기밀, 중층 상가 지구 2번 게이트, 위험: K-Sec 스캔]

출력:
키쿠모토 서버 팜 보안 분리 구역에서 기밀 코어가 하나 빠져나왔다는 제보가 들어왔다. 
현재 상가 지구 2번 게이트를 통해 반출되는 경로로 보이며, K-Sec이 해당 게이트에 
무선 식별 스캐너를 상시 가동 중인 것으로 관측됨. 
조용히 들어가서 조용히 꺼내 오면 된다. 아무도 몰라야 한다.
```

---

### 4.2 용병 이력서 생성

**입력:**

```json
{
  "merc_id": "merc_007",
  "name": "이름",
  "origin_faction": "더 밸브",
  "stats_tier": {
    "Frame": "중",
    "Cool": "하",
    "Wire": "상",
    "Cypher": "중",
    "Pulse": "중"
  },
  "equipment_summary": "사이버덱 탑재, 외장 무기 없음, 바이오 임플란트 중심",
  "system_tags": ["#넷러너", "#하층_출신"],
  "story_tags_hint": ["무언가를 찾고 있는 것 같다"],
  "current_status": "경미한 신경 피로",
  "analysis_level": 0
}
```

**출력 요구사항 (Phase별):**

| Phase | 출력 내용 |
|---|---|
| Phase 0 | 이름, 출신 암시, 이력 단서 2~3개, 현재 상태. 스탯 수치 절대 없음. |
| Phase 1 | 대표 역량 방향성 ("전자 장비 다루는 데 능숙하다"류), 장비 요약 |
| Phase 2 | 시스템 태그 상세, 만성 상태, 장비/임플란트 상세 |

> [!NOTE]
> 이력서는 특정 미션을 전제로 쓰지 않는다. "이 미션에 맞다"를 직접 말하지 않는다.  
> 행동 기록과 습관 중심으로 써서 플레이어가 판단하게 만든다.

### 4.3 결과 보고서 생성

**입력:**

```json
{
  "mission_id": "mission_001",
  "merc_id": "merc_007",
  "final_result": "성공",
  "node_results": [
    {"step": 1, "name": "보안망 스캔 및 우회", "result": "성공", "stat_used": "Cypher"},
    {"step": 2, "name": "내부 경비대 순찰망 회피", "result": "성공", "stat_used": "Cypher"},
    {"step": 3, "name": "기밀 데이터 무선 다운로드", "result": "성공", "stat_used": "Wire"},
    {"step": 4, "name": "침입 흔적 은폐 및 탈출", "result": "조건부 성공", "stat_used": "Cool"}
  ],
  "joker_triggered": false,
  "risk_node_triggered": false,
  "rewards": {
    "credits": 8000,
    "reputation_changes": [{"faction": "키쿠모토", "amount": -15}]
  },
  "status_changes": ["경미한 신경 피로 → 해소"],
  "equipment_changes": [],
  "story_hints": ["탈취한 기밀 안에 바이루이 연구 데이터가 섞여 있었다"]
}
```

**출력 요구사항:**

```
[사건 로그] (메인 읽을거리)
  - 각 노드를 시간 순서로 묘사. 수치 없이 상황으로.
  - 300~500자, 5~8문장.

[원인 확인] (짧은 요약)
  - 성공/실패 이유를 1~2문장으로.

[이야기 단서] (선택적)
  - story_hints가 있을 경우 마지막에 한 줄 암시.
```

> [!NOTE]
> 결과 로그는 부가 정보가 아니다. 게임의 핵심 읽을거리다.  
> "성공했습니다"가 아니라 어떻게 성공했는지, 뭔가 이상한 건 없었는지를 읽게 만든다.

## 5. 프롬프트 조립 순서 (생성 시)

```
[미션 브리핑 생성 기준]

1. [시스템 명령] Layer 1 정적 세계관 (항상 포함)
   → 도시 구조 + 팩션 압축 + 어조 규칙

2. [현재 상황] Layer 2 World State 스냅샷
   → 분쟁 중인 팩션 / 구역 알림 / 픽서 명성·악명

3. [생성 지시] Layer 3 미션 변수
   → 미션 뼈대 JSON

4. [출력 형식 지정]
   → Phase별 출력 형식 명시

5. [Few-shot 예시] (1~2개)
   → 좋은 출력 예시로 어조 고정
```

**총 토큰 예산 (Gemma 4 기준):**

| 구성 요소 | 목표 토큰 |
|---|---|
| Layer 1 정적 세계관 | ~300 |
| Layer 2 World State 스냅샷 | ~500 |
| Layer 3 생성 변수 | ~200 |
| 출력 형식 + Few-shot | ~300 |
| **총합** | **~1,300** |

> 전체 컨텍스트 2,048 기준 여유 확보. 출력 최대 200~500토큰.

## 6. AI 내러티브 품질 기준 (Stage 3 검증 기준)

### 6.1 미션 브리핑

| 기준 | 합격 조건 |
|---|---|
| 추론 가능성 | Phase 0 브리핑만 읽고 "어떤 성향의 용병이 필요할지" 방향이 잡히는가 |
| 세계관 정합성 | 팩션명, 구역명, 설정이 `static_data_spec.md`와 충돌하지 않는가 |
| 어조 일관성 | 하드보일드·냉소적 어조가 유지되는가 (친절한 설명 없음) |
| 수치 미노출 | 스탯명·태그명이 직접 언급되지 않는가 |

### 6.2 용병 이력서

| 기준 | 합격 조건 |
|---|---|
| 미션 정답표 아님 | "이 미션에 적합하다"는 표현이 없는가 |
| 행동 기록 중심 | 수치 나열이 아니라 습관·성향·사건으로 서술되는가 |
| 이야기 단서 포함 | Phase 0에서도 "뭔가 있다"는 뉘앙스가 암시되는가 |

### 6.3 결과 보고서

| 기준 | 합격 조건 |
|---|---|
| 사건 서술 우선 | "성공/실패" 결론보다 "어떻게 됐는지" 과정이 먼저 오는가 |
| World State 반영 | 현재 팩션 상태·픽서 명성이 내러티브에 자연스럽게 녹아드는가 |
| 후속 단서 | 다음 판단에 영향을 줄 이야기 단서가 포함되는가 |
