# 사이버펑크 용병관리소 — 프로젝트 전체 설명서

> 이 문서는 프로젝트 내 모든 기획 문서를 한 파일에 압축한 총괄 설명서다.  
> 원본 문서를 대체하지 않는다. 세부 규칙·수치·샘플은 각 폴더의 원문을 따른다.  
> 최신 기준: `00_프로젝트_관리/해야 할것.md` (2026-06-24 갱신)

---

## 0. 한 줄 정의

**픽서(플레이어)가 용병을 고용하고, 이력서·태그·스탯·장비·Visibility를 판독해 미션에 파견하는 텍스트 기반 부분 방치형 매니지먼트 게임.**

핵심은 태그 정답 맞추기가 아니라 **글을 읽고 픽서처럼 판단하는 경험**이다. 분석기와 Phase는 보조 장치이지, 플레이어 대신 정답을 말하는 장치가 아니다.

---

## 1. 핵심 플레이 경험

1. 미션 브리핑·용병 이력서의 **줄글**에서 성격, 장점, 위험, 필요 준비를 스스로 판독한다.
2. 5대 스탯, 태그, 장비 속성(Visibility), 임플란트로 용병/미션 적합성을 판단한다.
3. 스테이션으로 다음 1회 미션의 변수를 통제·해킹한다.
4. 파견 후 **결과 보고서 사건 로그**를 읽고 성공/실패 이유와 사건 전개를 다시 판독한다.
5. 보상, 상태, 평판, 관계 변화가 다음 판단을 바꾼다.

---

## 2. 게임 루프

```text
미션 게시판 확인
→ 용병 이력서 / 미션 카드 판독 (Phase 0~2)
→ 스탯·태그·장비·Visibility·임플란트 비교
→ 미션 슬롯 매칭 (min(용병 분석 Lv, 미션 분석 Lv))
→ 스테이션 준비 (선택)
→ 용병 파견 (어사인 / 캐치업)
→ 미션 노드 판정 (기본 관문 → 돌발 노드 → 조커 카드)
→ 결과 보고서 정산
→ 보상·상태·평판·후속 단서 반영
→ 다음 미션·성장 선택
```

**턴 구조:** 매일(1턴) 아침 신규 의뢰 생성. 미션마다 방치 시간(Decay) 존재 — 0이 되면 영구 소멸, 수주 후 방치 시 평판 패널티.

**파견 모드 (투트랙 — 2026-07-13 D-C-2):**
- **어사인:** 용병에 전부 맡김. `startDispatch` → 일괄 선계산 → Desk 타이머 → 용병 일지. Mid Risk / Mid Return.
- **캐치업:** 출격 후 **인런 드론 관제** — 노드마다 `[개입]`/`[패스]` 선택. 전용 상태 머신(`catchUpRun`). High Risk / High Return. 출격 전 노드 예약 체크박스는 **폐기**.

## 3. 플레이어 — 픽서

- 용병 고용·미션 수주·스테이션 운영·보수 배분·보험·장비/임플란트 제공.
- 팩션 평판, 계약 리스크, 용병 만족도 관리.
- 픽서 자체도 외형·집·임플란트·전문 업무·UI 테마 등 별도 성장 요소 보유(장기).
- 담당 구역별 의뢰 차별화 → 진영 선택 느낌.
- 랭킹(업무 처리·보수 효율), 만족도 기반 이벤트 트리거(장기).

> 주의: `02_핵심_시스템/픽서.md`는 픽서 설정이 아니라 **미션 5종·노드 판정** 원본 문서다.

## 4. 용병

미션에 파견되는 유닛. 보유 요소: **5대 스탯, 장비, 임플란트, 시스템 태그, 이야기 태그, 상태 태그, 이야기 단서.**

**태그 소속 (Step 0):** 용병 `systemTags` = 출신·행동·상태. 절연·은닉·스캐너 기만 등 **장비/임플란트 태그는 `entityTagLinks`(gear/implant)**. 런타임 `TriggeredTag`가 `sourceType`/`sourceId`로 출처를 구분.

**이력서 역할:** 상시 고용 정보. 특정 미션에 맞춘 정답표가 되면 안 된다. "이 미션에 적합"을 직접 말하지 않는다. 행동 기록·습관 중심 줄글로 성격·장점·위험을 읽게 한다.

**이력서 기본 필드:** 이름, 별칭, 출신, 고용 상태, 요구 보수, 5대 스탯, 대표 특기, 장비/임플란트 요약, 현재 상태, 이력 텍스트, 시스템 태그 후보, 이야기 단서 후보, 숨김 정보, 결과 보고서 연결 후보.

**용병 태그 초기 분류(레거시 참고):** 기본 태그(미션 특화·스탯 특화·이벤트 특화), 부정/긍정 태그(강화·비용·범죄/명성·상태·특징). 최신 기준은 `03_태그_사전/` 우선.

## 5. 5대 스탯

스탯은 태그가 아니다. `#낮은_Frame` 같은 스탯값 태그화 금지.

| 스탯 | 역할 |
|---|---|
| **Frame** | 육체·하드웨어 수용성·전투 생존·중화기. 고중량 크롬·전면전. |
| **Cool** | 침착·정신 마모 저항·협상·사이버싸이코 억제. 기업·비밀·위기 대응. |
| **Wire** | 네트워크·해킹·기계 조작·지원. 보안망·정밀 기기·지원 미션. |
| **Cypher** | 은닉·흔적 제거·현장 분석·잠입. 잠입·추적 회피·단서 분석. |
| **Pulse** | 직관·돌발 감지·비정형 위기. 비밀·변수 난무·기습 회피. |

태그는 스탯을 대체하지 않고, **어떤 상황에서 의미 있는지** 바꾼다.

## 6. 태그 시스템

### 6.1 정의

태그 = 용병·장비·미션·지역·팩션·상태·이야기를 압축하는 **짧은 정보 단위**. 장식 라벨이 아니다.

```text
태그 적용 = 대상 + 관계 + 해석
```

- **대상:** 용병, 장비, 임플란트, 미션, 지역, 팩션, NPC, 상태.
- **관계:** 출신, 보유, 요구, 위험, 대응, 평판, 소속, 노출, 계약.
- **해석:** 지금 미션/스토리/정보 단계에서 어떻게 읽히는가.

태그 자체는 단순. 복잡한 해석은 미션 규칙·이야기 규칙·정보 노출 규칙이 담당.

### 6.2 시스템 태그 vs 이야기 태그

| | 시스템 태그 | 이야기 태그 |
|---|---|---|
| **역할** | 미션 판정·장비·비용·상태·보상 | 캐릭터 서사·관계·비밀·장기 이벤트 |
| **읽히는 곳** | 진입 조건, 관문, 돌발, 조커, 시너지, 정산 | 이력서 암시, 미션 반응, NPC, 결과 단서, 후속 |
| **공개** | Phase 규칙 따름 | 분석기로 전부 공개 금지. Phase 2에서도 핵심 비밀 숨김 가능 |

**혼합 후보:** 하나로 뭉개지 않고 둘로 나눈다. 예) 신체 결함=시스템, 그 원인이 특정 팩션 실험=이야기.

### 6.3 상태 태그

시스템 태그 하위. 현재 컨디션·부상·피로·각성·경계심. 긍정·부정·중립·조건부 모두 포함.

변화는 수치 증가가 아니라 **신규 부여 / 악화·상위 전환 / 완화·하위 전환 / 해소·소모 / 병합·변질**.

- 임플란트 출력 저하·필터 오염 → 상태 태그가 아니라 **장비/임플란트 변화** 우선.
- 신체 증상(숨참, 기침, 떨림) → 결과 로그·증상 설명.
- 이번 미션에서 새로 생긴 상태 → Phase 무관 즉시 공개.

### 6.4 분류 체계 (10개 주 분류)

외형·가시성 / 성격·행동 / 출신·소속 / 전문·기술 / 무기 / 임플란트 / 상태·컨디션 / 미션·의뢰 / 지역·환경 / 이야기·관계

각 후보는 4축 판정: 시스템/이야기 × 공용/전용.

### 6.5 용병 특성 태그 사전 (1차)

6개 파일, 총 200개 수준: 작전환경(40), 소속팩션(40), 외형가시성(40), 장비크롬(30), 경제성향(25), 인물정신(25). 인물 중심·단어형·5:5~6:4 긍부정 비율.

### 6.6 변화 사전 (1차, 미확정)

- **상태 태그 사전:** 용병 신체/정신 컨디션.
- **장비 변화 사전:** 외장 무기·보호구·도구·소모품.
- **임플란트 변화 사전:** 내장형 무기·신체 파츠·보조 장기·신경계·연결구.
- **결과 정산 항목 사전:** 정비비·재시술비·보험·보상 항목.

원칙: 같은 판단을 만드는 세부 증상은 하나의 변화로 묶음. 부위별 차이는 변화명 늘리지 않고 `대상`·`부위별 적용`에서 처리.

### 6.7 태그 설계 금지

- 태그를 먼저 대량 생성하지 않는다.
- 스탯값을 태그화하지 않는다.
- Visibility를 태그가 아닌 **장비 속성/수치**로 분리.
- 긍정/부정 고정 효과 금지 — 미션이 상황별 해석.
- 공용 태그를 모든 곳에 막 붙이지 않는다.
- 지역/팩션 고유 구조는 전용 태그 규칙 재료.

### 6.8 코드 데이터 모델 (Step 0, 2026-06-24)

**3계층 저장 + 4계층 런타임:**

| 계층 | 저장 위치 | 역할 |
|---|---|---|
| TagDef | `tagDefinitions.ts` | 태그 속성만 (효과 없음) |
| EntityTagLink | `entityTagLinks.ts` | merc / gear / implant 부착 |
| TagMissionInterpretation | `tagMissionInterpretations.ts` | 맥락별 reading |
| TriggeredTag | 런타임 (`mercTagPool` → executor) | 발동 태그 + `sourceType`/`sourceId` |

**노드 극성:** `defaultPolarity` = adverse(디폴트 negative) | neutral(delta 0). `challengeTags`가 비어 있으면 태그 보정 없음.

**상태 머신:** `MissionRunExecutor` + `NodeQueue` + `MissionRunContext` + `RoutingDecision`. 검문·전기 등은 **고정 노드가 아니라** 생성기가 `challengeTags`를 붙일 때만 해당 기믹 발동. 상세: `코드_데이터_구조_온톨로지.md §3, §8, §10`.

## 7. 장비·임플란트·Visibility

### 7.1 장비 상위 개념

무기 + 임플란트. 미션 성공률, Visibility, 유지비, 파손 리스크에 영향. **텍스트 매니지먼트 요소** — 플레이어가 직접 조작하는 액션이 아님.

### 7.2 무기

| | 외장형 | 내장형 |
|---|---|---|
| 코스트 | 임플란트 코스트 미소모 | 코스트 소모 |
| 화력 | 높음, 바리에이션 다양 | 낮음 |
| Visibility | 높음 (검문·잠입·기업 위험) | ~0 (은닉) |
| 파손 | 수리비 | 임플란트 파손 → 막대 수리비 |
| 역할 | 교전 베이스 스탯 | 조커 카드·플랜 B |

**동시 장착:** 단순 합산 아님. Visibility=최대치 우선. 교전=Main(외장)+Sub(내장, 위기 시). 보안 적발 시 내장형=플랜 B.

**시너지:** 동일 `#태그` 명칭 일치 시 활성화. 불일치 시 깡스탯만.

**스탯 기반 무기군:** Frame(중화기), Wire(스마트/사이버덱), Cypher(은밀 정밀), Pulse(비정형·직감).

### 7.3 임플란트

**종류:** 바이오(저성능·자가회복·저비용) / 크롬(고성능·다양·고비용) / 기업·군용(최고급·최고비용·팩션 연결).

**주요 파츠:** 시각·청각·후각·구강 / 손·상완·하완 / 고관절·대퇴·하퇴·발 / 신경계·호흡계·순환계·소화계.

**출격 불가 조건:**
- 시각(맹인 빌드 제외), 하완, 하퇴/발(특수 이동 빌드 제외), 신경계(기본 칩셋) → 없거나 완전 파손 시 출격 불가.
- 호흡계 → 일반 미션 가능, 수중/독가스 등 특정 환경에서 강제 조건.

**스펙 설명문:** 제품 광고문 형태로 장단점 암시 → 플레이어 몰입·추론 유도.

### 7.4 Visibility

장비 속성/수치. 태그 아님. 검문·잠입·기업·보안 미션의 발각 가능성.

시각 노출뿐 아니라 발포음·작동음·열 배출·전자 신호·화학 흔적·군용/기업 마킹 포함. 높다고 항상 부정은 아님.

## 8. 미션 시스템

### 8.1 미션 타입 (문서 5종 / 구현 7종)

| 타입 | 핵심 | 성공 보상 | 실패 리스크 |
|---|---|---|---|
| **전투** | Frame, 생존, 화력 | 크레딧 다수 | 사망 |
| **잠입** | Cypher, Visibility, 은닉 | 정보·평판 | 체포 |
| **지원** | Wire, 기술·환경 대응 | 재화·재료 | 위약금 |
| **기업** | Cool, Cypher, Visibility, 팩션 | 기업 관계·장비·임플란트 | 기업 적대·사병 |
| **비밀** | Pulse, Cool, 미확인 위험 | 특수 재화·장비 | 모든 리스크+픽서 위험 |
| **교섭** | Cool, 협상·포섭·중재 | 평판·관계·정보 | 관계 악화·노출 |
| **추적** | Cypher, Pulse, 미행·회수·수색 | 회수물·단서 | 표적 도주·역추적 |

> 구현(`types.ts`)은 원래 `공작` 타입을 의미가 명확한 **`교섭`**(협상·포섭)과
> **`추적`**(미행·회수)으로 분리해 현재 7종이다. 미션 타입은 계층에서 배제되지 않고
> 가중치로만 차등 등장하며, 타깃 팩션은 **하강 규칙**(상위 계층 팩션은 하위 미션 대상 가능,
> 하위 계층 팩션은 상위 미션 대상 불가)을 따른다.

### 8.2 미션 카드

특정 용병을 전제로 하지 않는 **의뢰 정보**. 필드: 의뢰명, 타입, 위치, 팩션, 의뢰자, 목표, 보상, 위험도, 방치 시간, 브리핑, 기본 관문, 필수/권장/보너스 조건, 위험 노드, 조커 후보, 정산 후보, 이야기 단서.

Phase 0: 브리핑 줄글만. Phase 1: 신호. Phase 2: 시스템 요구 공개. 이야기 핵심 비밀은 Phase 2에서도 숨김 가능.

### 8.3 판정 파이프라인 (5단계)

```text
1. 진입 게이트 (필수 조건) — 미충족 시 출격 차단 또는 강제 위험 노드 [B-2 미착수]
2. 기본 관문 순차 해결 (3~7노드, NodeQueue) — 스탯+장비+태그(node_modifier)
3. 위험 노드 (돌발 함정) — RoutingDecision으로 inject / goto_phase
4. 조커 카드 개입 — 대가 교환(Trade-off) 필수
5. 결과 보고서·정산 — triggeredTags, nodeResolutions, seizedGearIds
```

**코드 (Step 0 완료):** `executeMissionRun`이 위 뼈대를 관장. 노드마다 `challengeTags` + `defaultPolarity`(adverse|neutral). **현 seed/AI 미션은 challenge 대부분 미부여 → neutral 플레이.** `missionTagMapping.ts` 가중 생성은 다음 스프린트.

**노드 수:** 난이도 1성=3노드, 1성 상승마다 +1.

**돌발 노드:** 기본 관문 대체/끼어듦. 부정(조건 부족·위험) / 긍정(권장 초과 충족·추가 보상) / 특수(이야기·팩션 반응).

**조커 카드 후보:** 서브 스탯, 내장 무기, 임플란트 특수 기능, 스테이션 1회 효과, 시스템/이야기 태그.

### 8.4 구역별 환경·생존 판정

| 구역 | 본질 | 기본 생존율 | 핵심 스탯 | 대표 위협 |
|---|---|---|---|---|
| **하층** (Tier 3) | 무법·독성·인프라 착취 | 60% | Frame, Pulse | 독성노출, 의체 부식, 정비비 |
| **중층** (Tier 2) | 갱단·K-Sec·추격 | 40% | Cypher, Cool | 수배, 교전, 가시성 폭증 |
| **상층** (Tier 1) | 기밀·바이오스캔·암살 | 20% | Cool, Cypher | 즉시 요격, 평판 붕괴, 청부 |

탈출 실패 시 **사생결단 생존 판정**: 구역 기본 생존율 + 핵심 스탯×가중치 + 적합 자질 - 약점 자질. 생존=의체 훼손 귀환, 실패=영구 사망·기어 유실.

## 9. 정보 해금 — Phase 0/1/2

**Phase = 추상 정보 강도.** 실제 해금은 스테이션 **이중 분석 기관** 레벨(Lv.0~2)이 결정.

| 기관 | Lv.0 | Lv.1 | Lv.2 |
|---|---|---|---|
| **용병 분석** | 이름·출신·이력 단서·현재 상태 | 5대 스탯 요약·장비/임플란트 요약 | 시스템 태그 상세·만성 상태 |
| **미션 분석** | 의뢰명·위치·보상·브리핑 | 추천 스탯 체급·위험 방향 신호 | 필수 기어·요구 수치·위협 프로필 |

**매칭 표시 레벨 = min(용병 분석 Lv, 미션 분석 Lv).** 한쪽만 높아도 매칭 정보 안 열림.

**매칭 Lv.1:** 유리/불리/조건부/미확인 + 거친 성공 전망.  
**매칭 Lv.2:** 충돌/시너지 이유, 예상 성공률, 정산 범위, 추가 베네핏.

**결과 보고서 Phase:** 두 분석 기관 교차 판정. 둘 중 하나라도 Lv.1 미만→Phase 0, 둘 다 Lv.1+ 하나라도 Lv.2 미만→Phase 1, 둘 다 Lv.2+→Phase 2.

**액티브 분석 슬롯:** 특정 대상 1턴 대기 후 분석 Lv +1. 미션 분석 슬롯 중 방치 시간 정지.

**예외:** 플레이어가 직접 만든 신규 상태=즉시 공개. 이야기 핵심 비밀=Phase 2에서도 숨김 가능.

> 주의: 정보 해금 Phase 0~2 ≠ 미션 판정 단계 1~4. 용어 혼동 금지.

## 10. 스테이션

픽서의 **단일 핵심 거점**. 5대 카테고리 중 **하나만** 선택. 영구 무적 보너스 없음 — **다음 1회 미션 변수 통제** 또는 리스크 해킹.

| 카테고리 | 역할 | 대표 시설 |
|---|---|---|
| **숙박** | 코어 스탯·환경 튜닝 | 보이드 생크텀(Cypher), 에이펙스 사나토리움(Frame), 마인드 팰리스(Cool), 코포레이트 서버룸(Wire) |
| **유흥** | 태그 제어·변수 창출 | 하이소사이어티(긍정 부여), 메가 클럽(부정 세탁), 하이롤러(펄스 도박) |
| **식사** | 임시 도핑·정보 | 시티 라운지(임플란트 버프), 하이테크(스탯 버프), 인텔리전스 허브(정보·소문) |
| **장비** | 하드웨어 공방 | 바이오테크(약물), 쉐도우 익스체인지(정크·유물), 전술 튜닝(Visibility 위장), 정밀 공학(시너지 강제·보험) |
| **업무** | 의뢰·보수 조작 | 섀도우 브로커(의뢰 리롤), 탤런트 매니지먼트(보수 배분·불만 억제) |

**분석 기능:** 미션 분석 Lv → 미션 카드·매칭·결과의 미션 조건 해석 깊이. 용병 분석 Lv → 이력서·매칭·결과의 용병 상태 해석 깊이.

**용병 서비스 및 고용소:** 무기·임플란트(탈착 가능)·보험(생명·장비)·식사·유흥·치료·성형 등의 확장이 예정되어 있으며, **현재 용병을 직접 크레딧으로 고용하고 해고하는 '용병 고용소' 기능이 구현 완료**됨. 스테이션 레벨업에 따른 지휘력(OP)을 기반으로 고용 명단을 관리.

## 18. 이 프로젝트가 만들려는 것 (요약)

사이버펑크 도시의 돔 안에서, 픽서는 용병 이력서와 미션 브리핑이라는 **텍스트**를 읽고 판단한다. 스탯과 태그, 장비의 Visibility, 임플란트 세팅으로 리스크를 관리하고, 스테이션으로 다음 한 번의 변수를 조작한다. 파견이 끝나면 결과 보고서의 사건 로그를 다시 읽으며 왜 그렇게 됐는지 납득하고, 상태·장비·평판의 변화가 다음 선택을 바꾼다.

게임의 재미는 정답을 맞히는 것이 아니라, **정보가 점진적으로 열리면서도 줄글의 판독 가치가 사라지지 않는 매니지먼트**에 있다. 현재는 그 루프가 문서·데이터·웹 프로토타입으로 1턴 검증 직전 단계에 있다.

## 19. AI 설계 마인드셋 (데이터베이스 정규화 우선)

프로젝트 코드 및 데이터 구조 변경 시 **반드시** 명심해야 할 핵심 마인드셋:

1. **상태 파이프라인(Status Pipeline) 활용 원칙**
   - 용병의 사망, 질병, 심리 상태 변화 등 새로운 결과가 필요할 때, `ResultReport` 같은 코어 객체에 임의의 변수(예: `mercSurvived: boolean`, `isSick: boolean`)를 마구잡이로 늘리지 않는다.
   - 이는 훗날 RDBMS 설계 시 불필요한 스키마 확장을 유발하는 **가장 나쁜 안티 패턴**이다.
   - 대신 이미 구축된 `statusChanges` (상태 변화 배열) 파이프라인을 활용한다. 예: "사망"이라는 결과를 추가하려면 `status_kia`라는 **상태(Status)** 하나를 추가하고 정산기(`settlement.ts`)가 이를 처리하게 한다.

2. **기존 데이터 구조의 재사용**
   - 새로운 기믹이나 패널티를 만들 때, 기존의 데이터 흐름(결과보고서 → 정산기 → 코어 상태)을 거스르지 않는지 철저하게 검증(Break & Structural Diagnosis)한 후 확장한다.
