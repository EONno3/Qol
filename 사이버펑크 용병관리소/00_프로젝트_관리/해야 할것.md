## ⚠️ 기획 vs MVP 구현 간극 (테크 부채 및 임시 시스템 해결 상태)

현재 Phase D(기초 구현 단계)의 'MVP 최소 작동 원칙'에 따라 단순화되었던 시스템들을 RDB 설계 정규화 및 기획 스펙에 맞추어 해결한 목록입니다.

### 1. 장비 및 인벤토리 시스템 (RDB 정규화 해결 완료)
- **기존 문제:** 장착 슬롯 및 소유 관계 부재. 장비 파손이 단순히 `mercStatuses`에 텍스트 상태 태그로 땜질되어 개별 장비 추적 불가능.
- **해결 조치:** `gearOwner` 및 `implantOwner` 관계 테이블(Record)을 도입하여 특정 용병이 어떤 장비를 소유하는지 명시적 릴레이션을 매핑. `replaceDestroyedGear`가 실제 파손된 장비 상태(`gearStates` 등)를 식별하여 `normal`로 리셋하도록 리팩토링 완료.

### 2. 조커 카드 및 태그 시너지 연동 (systemTags 해결 완료)
- **기존 문제:** 용병의 가시적 이력서 줄글(`phase2SummaryKo`)의 텍스트 매치(`includes`)에 의존해 판정하는 안티패턴.
- **해결 조치:** `Mercenary` 인터페이스 및 `seed.ts` 데이터에 실제 시스템 태그인 `systemTags: string[]` 필드를 추가하고, `shouldTriggerJoker` 및 `calculateGearDestructionProb`가 이 데이터를 직접 검색하도록 전면 교체 완료.

### 3. 사생결단 생존 굴림 연산 (태그 보너스 시너지 해결 완료)
- **기존 문제:** 태그 보정치가 `const tagBonus = 0;`으로 하드코딩되어 작동하지 않음.
- **해결 조치:** 용병의 `systemTags`를 기반으로 작전 구역(lower/mid/upper)별 긍정 자질(슬럼/유흥가/상층 적합 태그) 보유 시 `+15%` 생존율 보너스, 부정 자질(노출형 크롬 등) 보유 시 `-20%` 패널티가 동적으로 가산/감산되도록 구현 완료.

### 4. 로컬 AI 보고서 생성 (네트워크 병목 및 에러 제어 완료)
- **기존 문제:** 로컬 API 서버가 동작하지 않는 환경에서 백그라운드 fetch 실패 시 AbortError와 콘솔 경고가 폭증하여 웹 리소스 낭비 유발.
- **해결 조치:** `GAME_CONFIG.ai.aiNarratorEnabled` 옵션 및 우측 상태 패널 UI에 "로컬 AI 브리핑 활성화" 토글 체크박스를 연동. 비활성화 시 비동기 fetch를 가드하고 즉시 `FALLBACK` 결과로 자동 전환하도록 구현하여 불필요한 네트워크 오버헤드 완벽 차단.

### 5. 스테이션 시설 (5대 카테고리)
- **기획:** 숙박/유흥/식사/장비/업무 등 5대 카테고리별 고유 혜택과 해킹 요소 제공.
- **MVP 현황:** 시설 특색 없음. 단순히 돈을 내면 OP 최대치와 분석소 레벨만 올라가는 단일 '업그레이드 버튼'으로 구현.
- **고도화 목표:** 시설 카테고리별 특수 효과 구현 (예: 정보상 리롤, 태그 부정 세탁 등).

### 6. 캐치업 모드 (현장 개입)
- **기획:** 파견 시 1.5배의 코스트를 내고 픽서가 드론으로 30%의 노드에 직접 개입 (추가 보상 및 리스크 변동).
- **MVP 현황 (2026-07-09):** 토글·코스트×1.5·개입 -15%p·`catchUpBonusEarned`·**`extraRewardCredits`(보상 15% 가산)**·**D-C-1 출격 전 관제 드론 UI**(`DroneCatchUpView`, Glitch 패널) 완료.
- **후속 목표:** **D-C-2** 인런 실시간 드론 뷰(파견 실행 중 노드 진행).

### 7. 미션 생성 (Mission Generation)
- **기획:** AI 파이프라인이 층별, 팩션별 환경에 맞춰 텍스트 기반 미션을 무한 생성.
- **현황:** `batch_generate_missions.py`가 `src/data/generated_missions.json`(6개 미션)을 생성하고, `seed.ts`가 이를 import하여 게임에 반영하는 파이프라인 구축 완료. 현재 `generated_missions.json`에는 실제 AI가 생성한 브리핑 텍스트가 포함되어 있음. (MOCK 모드는 `.mock.json`으로 분리 저장하여 실 데이터 오염 방지.)
- **세계관 정합 완료 (2026-06-23):** 생성 풀(지명·팩션)과 AI 프롬프트를 돔 도시 '콜' 세계관으로 통일. 외부 IP(나이트시티·배드랜드·노매드 등) 및 '사이버펑크' 단어를 프롬프트 금지어로 차단. 미션 무대 '외곽' 삭제(시스템 타입 `LocationTier`도 하/중/상만). 계층별 군소 모임(`MINOR_GROUPS`) 슬롯 신설(현재 빈 풀, `MINOR_GROUP_SPAWN_CHANCE=0`). 재생성본 금지어 0건 검수 + 자동 테스트 136개 통과.
- **캐스케이딩 가중 선택 완료 (2026-06-24):** 미션 뼈대(장소→타입→타깃→난이도→의뢰인)를 조건부 가중치로 조립. 타깃 팩션은 하강 규칙(상위 계층 팩션은 하위 미션 대상 가능, 하위→상위 불가)을 준수. 미션 타입 `공작→교섭/추적` 분리(7종). 파이썬 단위 테스트 14개 통과.
- **추론 인프라 병렬화 완료 (2026-06-24):** 추론을 C++ `llama-server`(`-np 2` 연속 배치)로 분리하고 `narrator_server.py`를 raw `/completion` 프록시로 전환. 미션 생성과 보고서 narrate가 동시에 들어와도 슬롯에서 병렬 처리(검증 5.8s vs 직렬 8.1s). 실시간 미션 생성 도입의 기반.
- **플레이테스트 대기:** 브라우저 실제 구동 시 AI 브리핑 품질(팩션명 직접 노출 없이 정황 암시 여부), 미션 보드 표시, 결과 보고서 1인칭 일지, 콜 세계관 일관성 육안 확인 필요.
- **고도화 목표 (Phase E):** 구역별/타입별 미션 **실시간(런타임) 대량 생성** AI 파이프라인 연동(현재는 런처 배치). 군소 모임 항목 채우기 + 등장 활성화.

### 8. 테스트 밸런싱 상수 결합 부채 (해결 완료)
- **현상:** 초기 크레딧 및 레벨업 비용 등이 10배 이상 변경되었으나, 기존 테스트 코드가 구버전 상숫값을 하드코딩 검증하고 있어 테스트 파이프라인이 깨짐. 또한 시뮬레이션의 무작위성으로 인해 flaky 테스트 발생.
- **조치 (2026-06-16):** `src/data/config.ts` 파일로 모든 게임 밸런스 상수를 격리하여 테스트와 프로덕션 코드가 동일한 상수를 동적 참조하도록 리팩토링 완료. `App.test.tsx` 내의 주사위 무작위성도 Mocking 처리하여 Flaky 테스트 해결.

### 9. 코드 리뷰 리팩토링 (2026-06-18 완료)
- **내용:** 전체 코드베이스 정밀 리뷰 결과 발견된 5개 버그/타입 오류, 3개 하드코딩 잔재, 1개 중복 로직을 TDD+Tidy First 원칙으로 수정 완료.
- **해결 조치:**
  1. **GameState 이중 선언 제거** (`types.ts` 단일 정의, `state.ts`는 import re-export로 전환)
  2. **타입 명시화** (`LocationTier`, `GearStateValue` 타입 추가, `StationState.locationTier` 정확한 타입 적용)
  3. **`survival.ts` 중복 로직 제거** (동일 태그 보너스 계산 블록을 `getTagBonus()` 공통 함수로 추출)
  4. **비용 하드코딩 제거** (`station.ts`에 `getReplacementCost()` 추가, `StationView.tsx`의 15000 리터럴 제거)
  5. **밸런스 상수 단일화** (`GAME_CONFIG.difficulty`에 `statThresholdBase/PerStar`, `fatiguePerNodeByDifficulty` 추가, `engine.ts`/`dynamicSystem.ts` 모두 참조)
  6. **데드코드 제거** (`missionTypeNames` 영문→한글 맵 삭제, `missionTypeName()` pass-through로 단순화)
- **결과:** 테스트 89개 전체 통과 유지.

## MVP 우선 원칙 (변경 없음)

- 모든 시스템은 먼저 MVP에서 실제로 작동 가능한 최소 형태로 정의한다.
- 복잡한 고도화 규칙은 즉시 구현 대상으로 보지 않고, 고도화 후보로 분리한다.
- 문서 작성 시 MVP 필수 / 고도화 후보 / 보류 항목을 구분한다.
- 새 규칙이나 새 문서를 늘리기 전에 "이게 MVP에 필요한가?"를 먼저 확인한다.
