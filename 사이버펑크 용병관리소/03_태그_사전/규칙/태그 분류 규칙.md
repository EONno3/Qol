# 태그 분류 규칙

이 문서는 태그 후보를 어떤 범주에 넣을지 정하는 기준 문서다.

실제 태그 목록을 작성하지 않는다. 태그 후보가 나왔을 때 공용/전용, 시스템/이야기, 주 분류를 판단하는 방법을 정한다.

## 1. 분류 목적

태그 분류의 목적은 태그를 많이 나누는 것이 아니다.

목적:

- 플레이어가 읽기 쉬운 구조 유지.
- AI가 태그를 만들 때 중복 방지.
- 시스템 태그와 이야기 태그 혼선 방지.
- 공용 태그와 전용 태그 오분류 방지.
- 미션/이력서/장비/팩션 문서 연결 명확화.

## 2. 1차 판정

모든 태그 후보는 먼저 아래 네 질문으로 판정한다.

```text
1. 시스템 태그인가?
2. 이야기 태그인가?
3. 공용 태그인가?
4. 전용 태그인가?
```

시스템/이야기, 공용/전용은 별개 축이다.

예:

```text
공용 시스템 태그
전용 시스템 태그
공용 이야기 태그
전용 이야기 태그
```

## 3. 주 분류 10개

태그 후보는 아래 10개 중 하나를 주 분류로 가진다.

### 1. 외형·가시성

인물, 장비, 임플란트, 무장의 눈에 띔과 사회적 통과 가능성을 다룬다.

사용 판단:

- 검문.
- 잠입.
- 의전.
- 위장.
- 공포/신뢰 인상.

### 2. 성격·행동

용병/NPC/팩션의 행동 패턴, 성향, 문제 해결 방식을 다룬다.

사용 판단:

- 이력서 판독.
- 협력/배신 가능성.
- 위기 반응.
- 이야기 이벤트.

### 3. 출신·소속

지역, 팩션, 계층, 문화권, 신분 연결을 다룬다.

사용 판단:

- 지역 적응.
- 팩션 반응.
- 평판.
- 통행/검문.
- 의뢰 신뢰도.

### 4. 전문·기술

용병이나 NPC가 어떤 일을 잘하는지 다룬다.

사용 판단:

- 미션 역할.
- 지원/해킹/수리/의료/운송.
- 특정 노드 돌파.

### 5. 무기

외장형/내장형 무기, 공격 방식, 화력, 가시성, 플랜 B를 다룬다.

사용 판단:

- 전투력.
- 은닉성.
- 검문 리스크.
- 조커 카드.
- 파손/보험.

### 6. 임플란트

신체 파츠, 코스트, 환경 대응, 신경계, 바이오/크롬/군용 차이를 다룬다.

사용 판단:

- 출격 조건.
- 스탯 보정.
- 환경 생존.
- 장비 시너지.
- 파손/수리비.

### 7. 관계·평판

인물, 팩션, 지역 사이의 우호/적대/채무/신뢰를 다룬다.

사용 판단:

- 미션 수주.
- 보상.
- 후폭풍.
- 배신/보호.
- 수배.

### 8. 계약·경제

고용비, 보수 배분, 보험, 유지비, 채무, 재화 보상을 다룬다.

사용 판단:

- 픽서 경영.
- 장비 보험.
- 용병 불만.
- 스테이션 비용.
- 미션 보상.

### 9. 상태·후유증

부상, 피로, 스트레스, 파손, 신분 노출, 만성 문제를 다룬다.

사용 판단:

- 다음 미션 제약.
- 즉시 공개 여부.
- 치료/수리/휴식.
- 돌발 노드.

### 10. 이야기·비밀

과거, 비밀, 장기 이벤트, 특정 인물 관계, 반전 정보를 다룬다.

사용 판단:

- 후속 이벤트.
- NPC 반응.
- 스토리 분기.
- 숨김 정보.

## 4. 주 분류 선택 원칙

태그 후보가 여러 분류에 걸치면 하나만 주 분류로 선택한다.

선택 기준:

```text
1. 플레이어가 그 태그를 보고 가장 먼저 판단해야 하는 것.
2. 미션이 가장 먼저 읽는 위치.
3. UI에서 보여줄 섹션.
4. 결과 보고서에서 정산되는 위치.
```

나머지는 보조 분류로 기록할 수 있다.

## 5. 중복 방지 규칙

같은 의미의 태그를 이름만 다르게 만들지 않는다.

금지:

- 외형 태그와 가시성 태그가 같은 의미인데 따로 존재.
- 출신 태그와 소속 태그가 같은 정보를 반복.
- 시스템 태그와 이야기 태그가 같은 내용을 중복 기록.
- 공용 태그와 전용 태그가 같은 판단 질문을 동시에 가짐.
- 스탯 수치/등급을 태그로 다시 만드는 것.
- `#낮은_Frame`, `#높은_Cypher`, `#Wire_경계값`처럼 스탯 상태를 태그명으로 만든 것.

필요하면 하나를 기준으로 두고, 다른 쪽은 참조 관계로 둔다.

스탯 처리 기준:

- 스탯은 태그가 아니라 용병 기본 수치다.
- 스탯 부족/초과는 매칭 화면에서 `Frame 36 → 요구 60 미충족`처럼 판정한다.
- 스탯과 별개로 존재하는 경험, 성향, 상태만 태그가 될 수 있다.

## 6. 혼합 후보 처리

한 태그 후보가 여러 기능을 동시에 가지면 쪼갠다.

```text
시스템 판단 부분
이야기 이유 부분
외형 표시 부분
경제 비용 부분
```

이렇게 나눠야 미션이 읽기 쉽다.

## 7. 공용/전용 분류

### 공용으로 올릴 조건

- 여러 대상에서 같은 판단 질문을 만든다.
- 특정 고유명 없이도 작동한다.
- 미션 해석 규칙에서 반복 사용된다.
- 플레이어가 직관적으로 이해할 수 있다.

### 전용으로 내릴 조건

- 특정 지역, 팩션, 장비, 인물군에서만 의미가 선명하다.
- 공용으로 올리면 세계관 고유성이 죽는다.
- 같은 단어라도 대상에 따라 의미가 심하게 갈린다.
- 특정 문서의 설정 재료에 가깝다.

## 8. 시스템/이야기 분류

### 시스템 태그로 분류

- 출격/성공률/노드/비용/보상/상태에 직접 연결된다.
- Phase 2에서 충돌 이유를 공개할 수 있다.
- 결과 보고서에서 정산된다.

### 이야기 태그로 분류

- 과거/관계/비밀/장기 이벤트에 연결된다.
- 분석기로 전부 공개하면 재미가 줄어든다.
- 특정 NPC/팩션 반응을 만든다.
- 후속 이벤트를 연다.

## 9. UI 분류

이력서와 미션 카드에서 보여줄 때는 분류를 줄인다.

기본 UI 분류:

```text
스탯
장비/임플란트
대표 분석
현재 상태
미확인 위험
이야기 단서
```

내부 분류 10개를 UI에 그대로 전부 노출하지 않는다.

화면 표시량은 `이력서 및 미션 카드 압축 규칙.md`를 따른다.

## 10. 검토 질문

태그 후보마다 아래를 확인한다.

```text
1. 주 분류는 무엇인가?
2. 시스템 태그인가, 이야기 태그인가?
3. 공용인가, 전용인가?
4. 다른 태그와 의미가 겹치지 않는가?
5. 미션이 읽을 수 있는가?
6. 이력서나 미션 카드에서 보여줄 필요가 있는가?
7. Phase별 공개 규칙을 어기지 않는가?
```
