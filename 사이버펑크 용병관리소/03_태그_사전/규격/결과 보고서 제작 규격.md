# 결과 보고서 제작 규격

이 문서는 실제 결과 보고서를 만들기 위한 1차 제작 규격이다.

최종 문체 확정 문서가 아니다. 목적은 미션 카드, 용병 이력서, 미션 슬롯 매칭 이후 **파견 결과를 어떻게 정산하고 다음 선택으로 연결할지** 정하는 것이다.

## 1. 문서 목적

결과 보고서는 플레이어에게 다음 질문에 답해야 한다.

```text
무슨 일이 일어났는가?
왜 성공/실패/부분 성공이 되었는가?
어떤 조건이 충족되었고, 무엇이 부족했는가?
용병에게 어떤 상태 태그 변화가 생겼는가?
장비/임플란트/보험/보상/평판은 어떻게 정산되었는가?
다음 선택에 어떤 영향이 생겼는가?
```

결과 보고서는 추론용 브리핑이 아니다.

플레이어가 직접 보낸 미션에서 실제로 발생한 일을 정산하는 화면이다.

동시에 결과 보고서는 다음 판단을 위한 텍스트 판독 자료다.

플레이어는 결과 로그를 읽고 “왜 성공했는지”, “왜 손실이 났는지”, “다음에는 무엇을 준비해야 하는지”를 스스로 판단해야 한다.

## 2. 기본 원칙

- 결과 보고서는 파견 전 정보가 아니라 파견 후 실제 발생 사실을 보여준다.
- 상태 태그 변화, 부상, 장비 파손, 임플란트 손상, 수배/노출은 숨기지 않는다.
- 성공/실패 이유는 플레이어가 납득할 수 있게 설명한다.
- 모든 내부 계산식을 공개하지는 않는다.
- Phase가 낮아도 플레이어가 직접 만든 결과는 명확히 표시한다.
- 기존부터 숨겨져 있던 만성/내재 문제는 결과에서 증상이나 사건으로 드러날 수 있다.
- 이야기 핵심 비밀은 결과 보고서에서도 전부 공개하지 않을 수 있다.
- 결과 보고서는 용병 이력서, 미션 카드, 미션 슬롯 매칭, 다음 미션 목록에 영향을 돌려보내야 한다.
- 결과 보고서도 Phase별 표시 구조를 가진다.
- 상태 태그 변화는 이름만 표시하지 않고 원인, 효과, 해소 조건을 함께 설명한다.
- 보상과 재화는 임시값이라도 수치로 표시한다.
- 결과 로그는 단순 장식이나 접기용 부가 정보가 아니라 핵심 플레이 판단 재료다.
- 실패/성공 이유를 정답표처럼만 나열하지 않고, 사건 전개를 통해 플레이어가 읽어낼 수 있게 쓴다.
- 분석 정보는 결과 로그를 대체하지 않고, 로그를 읽은 뒤 확인하는 보조 정보로 둔다.
- 플레이어 표시 순서는 기본적으로 `사건 로그 → 원인 신호/판정 확인 → 정산` 흐름을 따른다.
- 기존 상태와 이번 미션으로 생긴 상태 변화는 분리해 표시한다.
- 플레이어용 후속 단서와 내부 시스템 플래그는 같은 문장에 섞지 않는다.
- 손실과 추가 베네핏 미획득은 분리한다.
- Phase가 올라가면 같은 회수물이나 단서도 더 정확한 이름으로 해석될 수 있다.

## 3. 결과 보고서 기본 필드

결과 보고서는 아래 필드를 가진다.

```text
미션명
투입 용병
최종 결과
요약 로그
핵심 판정 요약
조건 충족/미충족
조커 카드 발동
돌발 노드 발생
보상
추가 베네핏 미획득
손실
상태 태그 변화
상태 태그 효과
장비/임플란트 변화
평판/수배/노출 변화
이야기/후속 단서
다음 반영 위치
```

모든 필드가 항상 같은 무게로 보일 필요는 없다.

## 4. 내부 작성 양식

결과 보고서를 만들 때는 먼저 내부용으로 전부 적는다.

이 내부 양식은 플레이어에게 그대로 보여주지 않는다.

```text
## 결과 보고서명

### 기본 정보
- 미션명:
- 투입 용병:
- 미션 타입:
- 최종 결과: 성공 / 부분 성공 / 실패 / 대가 있는 성공 / 사고
- 예상 성공 전망:
- 실제 결과 차이:

### 주요 판정
- 충족한 필수 조건:
- 미충족한 필수 조건:
- 충족한 권장 조건:
- 미충족한 권장 조건:
- 획득한 추가 베네핏:
- 실패한 추가 베네핏:

### 진행 로그
- 기본 관문 결과:
- 돌발 노드:
- 조커 카드:
- 상태/장비 반응:
- 이야기 태그 반응:

### 정산
- 기본 보상:
- 추가 보상:
- 손실/위약금:
- 장비/임플란트 파손:
- 상태 태그 변화:
- 상태 태그 효과:
- 평판 변화:
- 수배/노출 변화:
- 후속 단서:

### 다음 반영
- 용병 이력서 현재 상태에 반영:
- 장비/임플란트 상태에 반영:
- 팩션 평판에 반영:
- 미션 목록/후속 의뢰에 반영:
- 이야기 이벤트 후보:
```

## 5. 플레이어 노출 구조

### 5.1 요약 영역

목적:

- 플레이어가 결과를 5초 안에 파악한다.

보여줄 것:

```text
미션명
투입 용병
최종 결과
핵심 보상/손실
상태 태그 변화
```

예:

```text
결과: 부분 성공
보상: 크레딧 +1,200, The Valve 평판 +1
손실: 정비비 -180, #피로누적 → #심각한_피로누적
임플란트 변화: #호흡보조_불안정 → #호흡보조_기능저하
```

### 5.2 핵심 판정 영역

목적:

- 사건 로그를 읽은 뒤 왜 그런 결과가 나왔는지 확인한다.

보여줄 것:

- 충족한 핵심 조건.
- 부족했던 조건.
- 성공을 만든 요소.
- 손실을 만든 요소.
- 조커 카드 발동 여부.

작성 규칙:

- 핵심 판정은 결과 로그를 대체하지 않는다.
- Phase 1/2에서도 사건 로그를 유지하고, 분석은 로그 뒤에 붙인다.
- 미션 슬롯 매칭에서 보여준 예측과 연결되어야 한다.
- 결과가 예측과 달라졌다면 이유를 남긴다.
- 스탯 부족/초과는 태그가 아니라 수치/판정으로 설명한다.
- `Visibility`는 태그가 아니라 장비 속성으로 설명한다.

### 5.3 로그 영역

목적:

- 결과를 사건 전개로 보여준다.
- 플레이어가 다음 판단에 쓸 성공/실패 원인을 읽어낼 수 있게 한다.

보여줄 것:

- 주요 장면 3~6개.
- 기본 관문 통과/실패.
- 돌발 노드 발생.
- 조커 카드 발동.
- 용병 성향이나 이야기 태그가 반응한 순간.

작성 규칙:

- 로그는 결과의 이유를 설명해야 한다.
- 단순 분위기 문장만 늘리지 않는다.
- 로그는 플레이어가 읽고 판단하는 핵심 텍스트이므로 기본 판단 흐름에서 배제하지 않는다.
- 장면이 길어질 경우 본문에는 핵심 원인과 전환점을 남기고, 반복 묘사나 세부 전투 동작만 보조 영역으로 보낸다.
- “이 태그 때문에 실패” 같은 해답형 문장만 쓰지 말고, 그 태그가 어떻게 사건으로 드러났는지 쓴다.
- 플레이어용 진행 로그에는 태그명, 스탯 수치, 권장 기준 표기를 직접 섞지 않는다.
- 태그명, 스탯 수치, 권장 기준은 로그 뒤의 원인 신호/정확한 판정 영역에서 확인하게 한다.

### 5.4 정산 영역

목적:

- 다음 플레이에 영향을 주는 변화를 명확히 표시한다.

보여줄 것:

- 보상.
- 추가 베네핏.
- 추가 베네핏 미획득.
- 위약금/손실.
- 상태 태그 변화.
- 장비/임플란트 변화.
- 평판/수배/노출 변화.
- 후속 의뢰/단서.

작성 규칙:

- 상태 태그 변화와 파손은 반드시 명확히 표시한다.
- 보상과 손실을 같은 영역에서 비교 가능하게 둔다.
- 이미 가진 자원, 장비, 상태, 평판이 깎이면 손실로 둔다.
- 얻을 수 있었지만 얻지 못한 보너스, 회수품, 추가 보상은 추가 베네핏 미획득으로 둔다.
- 다음 화면에 반영될 항목을 표시한다.
- 보상, 비용, 재화, 평판은 임시값이라도 수치로 표시한다.
- 수치가 확정 전이면 문서 상단에 더미 수치임을 명시한다.

### 5.5 상태 태그 설명 영역

목적:

- 새로 붙거나 변화한 상태 태그가 다음 선택에 어떤 영향을 주는지 설명한다.

보여줄 것:

```text
상태 태그
변화 유형
원인
효과
해소/완화 조건
다음 반영 위치
```

작성 규칙:

- 범용 상태는 이름만 쓰지 않는다.
- 상태는 스탯이 아니라 상태 태그로 처리한다.
- `피로 +1단계`처럼 수치/단계가 있는 것처럼 쓰지 않는다.
- 기존부터 있던 상태 태그와 이번 미션으로 새로 생긴 상태 태그를 구분한다.
- 기존 상태가 변화한 경우 `기존 태그 → 변화 후 태그` 형태로 쓴다.
- 효과는 특정 미션 불리, 비용 증가, 휴식 필요, 일시 보정/역보정 등으로 설명한다.
- 정확한 수치가 미정이면 `다음 1회`, `장시간 작업`, `환경 미션` 같은 범위형 설명을 쓴다.
- 상태가 숨겨진 만성 문제와 연결될 수 있어도, 결과 보고서에서는 새로 발생한 상태 자체를 숨기지 않는다.
- 신체 상태, 장비 상태, 임플란트 기능 저하를 한 상태 태그로 뭉개지 않는다.
- 임플란트 출력 저하, 필터 오염, 부품 손상은 `장비/임플란트 변화`로 우선 처리한다.
- 그로 인해 드러난 기침, 숨참, 떨림 같은 증상은 로그나 부가 증상 설명으로 둘 수 있다.

### 5.6 상태 태그 변화 유형

상태 태그는 부정 상태만 의미하지 않는다.

컨디션이 좋은 상태, 안정 상태, 각성 상태, 회복 상태, 부상, 피로, 오염, 스트레스처럼 파견 전후 판단에 영향을 주는 현재 조건을 모두 포함한다.

기본 변화 유형:

```text
신규 부여: 없음 → #긴장
악화: #피로누적 → #심각한_피로누적
완화: #심각한_피로누적 → #피로누적
해소: #피로누적 → 제거
상위 전환: #양호한_컨디션 → #최상의_컨디션
하위 전환: #최상의_컨디션 → #양호한_컨디션
소모: #양호한_컨디션 → 제거
병합: #피로누적 + 추가 피로 압력 → #심각한_피로누적
변질: #각성 → #흥분
```

작성 규칙:

- 긍정 상태도 미션 결과로 생기거나 사라질 수 있다.
- 긍정 상태도 모든 미션에서 좋은 것은 아니다.
- 부정 상태도 모든 미션에서 즉시 실패를 뜻하지 않는다.
- 상태 태그의 유불리는 미션 조건이 해석한다.
- 상태 태그 변화는 가능한 한 태그 추가/제거/전환으로 표현한다.
- 상태 태그명은 플레이어가 효과를 직관적으로 추론할 수 있어야 한다.

## 6. 결과 유형

### 성공

목표를 달성하고 핵심 손실이 없다.

가능한 정산:

- 기본 보상.
- 평판 상승.
- 추가 베네핏 일부.
- 긍정 상태 유지 또는 경미한 상태 소모.

### 부분 성공

주 목표는 달성했지만 손실이나 미달 조건이 있다.

가능한 정산:

- 기본 보상 일부 또는 전부.
- 추가 베네핏 실패.
- 상태 태그 변화.
- 장비 오염/파손.
- 평판 상승과 불신이 동시에 발생.

### 실패

주 목표를 달성하지 못했다.

가능한 정산:

- 보상 없음 또는 위약금.
- 평판 하락.
- 부상/파손/노출.
- 후속 복구 미션 발생.

### 대가 있는 성공

목표는 달성했지만 이후 리스크가 커진다.

가능한 정산:

- 보상 획득.
- 수배/노출 증가.
- 특정 팩션 적대.
- 용병 충성도/스트레스 변화.

### 사고

미션 결과와 별개로 큰 돌발이 발생한다.

가능한 정산:

- 상태 태그 변화.
- 장비/임플란트 큰 손상.
- 이야기 태그 공개 후보.
- 후속 이벤트 강제 발생.

## 7. 스테이션 분석 레벨과 결과 보고서 Phase

결과 보고서 Phase는 독립적으로 해금되는 별도 레벨이 아니다.

결과 보고서는 파견에 사용된 **미션 분석 기능 레벨**과 **용병 분석 기능 레벨**을 함께 참조해 표시 깊이를 정한다.

```text
미션 분석 기능 레벨: 미션 조건, 현장 위험, 보상, 돌발 변수 해석력
용병 분석 기능 레벨: 용병 상태, 태그, 장비, 만성 문제 해석력
```

결과 보고서는 정보 해금 Phase와 관계없이 발생 사실을 숨기지 않는다.

다만 왜 그런 결과가 나왔는지 해석하는 깊이는 두 분석 기능의 조합에 따라 달라진다.

### 결과 보고서 Phase 판정 기준

| 조건 | 결과 보고서 Phase | 표시 기준 |
|---|---|---|
| 미션 분석 Lv.1 미만 또는 용병 분석 Lv.1 미만 | Phase 0 | 발생 사실, 보상, 손실, 상태 태그 변화 중심 |
| 미션 분석 Lv.1 이상 + 용병 분석 Lv.1 이상, 단 둘 중 하나라도 Lv.2 미만 | Phase 1 | 대표 원인 신호와 유리/불리 방향 표시 |
| 미션 분석 Lv.2 이상 + 용병 분석 Lv.2 이상 | Phase 2 | 정확한 조건 충돌, 스탯 판정, 태그/장비 속성 반응 표시 |

결과 보고서 샘플을 만들 때는 가능하면 Phase 0/1/2 표시를 실제 화면 형태로 나눈다.

단순히 "Phase별 차이"를 설명하는 보조 섹션만 두면 부족하다.

#### Phase 1 내부 표현 차이

미션 분석 레벨과 용병 분석 레벨이 어긋난 경우에도 별도 Phase로 나누지 않는다.

둘 중 하나가 Lv.1이고 다른 하나가 Lv.2라면 결과 보고서는 Phase 1이다.

다만 Phase 1 안에서 어느 쪽 정보가 더 선명한지 표현 차이가 생길 수 있다.

예:

```text
미션 분석 Lv.2 / 용병 분석 Lv.1
→ 현장 조건은 비교적 정확히 보이나, 용병 내부 상태와 태그 반응은 신호 수준으로만 보인다.

미션 분석 Lv.1 / 용병 분석 Lv.2
→ 용병 상태는 비교적 정확히 보이나, 어떤 미션 조건과 충돌했는지는 신호 수준으로만 보인다.
```

### Phase 0

- 상태 태그 변화, 보상, 파손, 평판 변화는 직접 표시한다.
- 숨겨진 만성/내재 문제의 정확한 이름은 공개하지 않을 수 있다.
- 이유는 현상 중심으로 쓴다.
- 보상/손실 수치는 표시한다.
- 미션 분석 또는 용병 분석 중 하나라도 Lv.1에 도달하지 못한 상태다.

예:

```text
터널 진입 중 호흡이 불안정해 작업 시간이 늘어났다.
```

### Phase 1

- 대표 원인 신호를 표시한다.
- 정확한 숨김 태그명은 여전히 감출 수 있다.
- 상태 효과는 표시하되, 숨김 태그명과 내부 계산식은 감춘다.
- 미션 분석과 용병 분석이 모두 Lv.1 이상이지만, 둘 중 하나라도 Lv.2에 도달하지 못한 상태다.

예:

```text
불리 신호: 호흡계 상태와 환경 조건 충돌.
```

### Phase 2

- 시스템상 필요한 충돌 이유를 명확히 공개한다.
- 숨겨진 만성 상태나 태그가 결과에 직접 영향을 줬다면 공개 가능하다.
- 이야기 핵심 비밀은 여전히 일부 숨길 수 있다.
- 조건 충족/미충족, 스탯 판정, 장비 속성 판정을 명확히 표시한다.
- 미션 분석과 용병 분석이 모두 Lv.2 이상인 상태다.

예:

```text
충돌: #호흡보조_불안정 → 유독 환경 장시간 체류 위험 증가.
```

## 8. 상태 태그와 이력서 반영

결과 보고서에서 새로 발생하거나 변화한 상태 태그는 용병 이력서 현재 상태에 즉시 반영한다.

반영 대상:

- 신규 부상.
- 피로/스트레스.
- 장비 파손.
- 임플란트 손상.
- 오염.
- 수배/노출.
- 팩션 경계.

작성 규칙:

- 플레이어가 만든 결과는 숨기지 않는다.
- 상태 태그와 변화 유형은 명확히 표시한다.
- 원인, 효과, 해소/완화 조건을 함께 남긴다.
- 상태 효과가 아직 수치화되지 않았다면 적용 범위를 문장으로 설명한다.

예:

```text
상태 태그 변화: #피로누적 → #심각한_피로누적
원인: 장시간 작업과 회복 부족
효과: 다음 1회 장시간/정밀 작업에서 불리
완화: 휴식, 식사, 스테이션 회복
다음 반영: 용병 이력서 현재 상태에 즉시 표시
```

## 9. 장비/임플란트 정산

장비/임플란트 변화는 결과 보고서에서 명확히 표시한다.

정산 후보:

- 오염.
- 내구도 감소.
- 파손.
- 수리 필요.
- 보험 적용.
- 영구 손실.
- 임플란트 손상.

작성 규칙:

- 외장형 장비와 내장형 장비를 구분한다.
- 임플란트 손상은 용병 상태와 비용에 함께 영향을 줄 수 있다.
- 보험이 개입했다면 비용 감소/복구 가능 여부를 표시한다.
- `Visibility` 때문에 생긴 결과는 장비 속성 반응으로 쓴다.
- 임플란트 기능 저하를 신체 상태 태그로 대체하지 않는다.
- 임플란트 기능 저하가 신체 증상을 만들면, 기능 변화와 증상 설명을 분리한다.

## 10. 보상/평판/후속 단서

결과 보고서는 다음 미션 선택을 바꿔야 한다.

정산 후보:

- 크레딧.
- 플럭스.
- 크롬.
- 장비 할인.
- 팩션 평판.
- 수배/노출.
- 후속 의뢰.
- 은폐 기록.
- NPC 반응.

작성 규칙:

- 보상은 획득/미획득/부분 획득을 구분한다.
- 손실과 추가 베네핏 미획득을 혼동하지 않는다.
- `침입자 장비 회수 실패`처럼 얻지 못한 선택 보너스는 손실이 아니라 추가 베네핏 미획득으로 처리한다.
- `정비비 -180 크레딧`, `소형 비살상 도구 1회 소모`처럼 실제 자원이 줄어든 항목은 손실로 처리한다.
- 보상과 재화는 수치로 표시한다.
- 수치가 확정 전이면 더미 수치로 표시하고 확정값처럼 취급하지 않는다.
- 평판 변화는 어느 팩션에 적용되는지 표시한다.
- 후속 단서는 너무 노골적인 정답명으로 쓰지 않는다.
- 같은 단서라도 Phase 0에서는 조각/흔적 수준, Phase 1에서는 의심 로그 수준, Phase 2에서는 정확한 기록명 수준으로 표시할 수 있다.
- `즉시 후속 미션은 열리지 않음` 같은 내부 시스템 설명은 보고서 본문에 직접 쓰지 않는다.
- 후속 단서는 플레이어가 나중에 연결 가능성을 느끼는 정도로 둔다.
- `경계 후보`, `이벤트 후보`, `후속 미션 플래그` 같은 내부 상태명은 플레이어 보고서 본문에 직접 쓰지 않는다.
- 내부 플래그가 필요하면 별도 내부 정산 영역에 둔다.

## 11. 이야기 태그와 결과 보고서

이야기 태그는 결과 보고서를 풍부하게 만들 수 있다.

하지만 결과 보고서가 모든 비밀을 공개하는 화면은 아니다.

사용 방식:

- 이상 반응.
- 특정 팩션/NPC 반응.
- 용병의 침묵, 회피, 분노, 집착.
- 후속 이벤트 단서.
- 이력서 확장 후보.

작성 규칙:

- 이야기 핵심 비밀은 필요할 때까지 숨길 수 있다.
- 결과 보고서에서는 "무언가 있다"는 단서만 줄 수 있다.
- 시스템 정산과 이야기 암시를 한 문장에 섞지 않는다.

## 12. 결과 보고서 금지 사항

- 신규 부상이나 파손을 숨기지 않는다.
- 실패 이유를 아무 설명 없이 `운이 나빴다`로 처리하지 않는다.
- 스탯값을 태그처럼 쓰지 않는다.
- `Visibility`를 부정 태그처럼 쓰지 않는다.
- 이력서에 들어가야 할 상시 인물 정보를 결과 보고서에 새로 꾸며 넣지 않는다.
- 미션 카드에 없던 핵심 위험을 결과에서 갑자기 정답처럼 꺼내지 않는다.
- 모든 내부 계산식을 그대로 공개하지 않는다.
- 이야기 핵심 비밀을 결과 보고서 하나로 전부 소모하지 않는다.
- 결과 로그를 단순 장식이나 접기 전용 부가 정보로 취급하지 않는다.
- 성공/실패 이유를 사건 전개 없이 판정표만으로 처리하지 않는다.
- Phase가 올라갔다고 사건 로그를 판정표로 대체하지 않는다.
- 기존 상태 태그와 이번 미션의 태그 변화를 같은 목록에 뭉뚱그리지 않는다.
- 내부 플래그명을 플레이어용 후속 단서처럼 노출하지 않는다.

## 13. 작성 완료 체크리스트

결과 보고서 하나를 만든 뒤 아래를 확인한다.

- 최종 결과가 명확한가?
- 성공/실패/부분 성공 이유가 설명되는가?
- 미션 슬롯 매칭에서 예측한 위험과 연결되는가?
- 상태 태그 변화와 파손이 즉시 공개되는가?
- 보상과 손실이 구분되는가?
- 손실과 추가 베네핏 미획득이 구분되는가?
- 보상과 재화가 수치로 표시되는가?
- 상태 태그 변화의 원인, 효과, 해소/완화 조건이 표시되는가?
- 평판/수배/후속 단서가 다음 선택으로 이어지는가?
- 후속 단서가 정답명이나 내부 시스템 설명으로 노출되지 않는가?
- Phase별로 후속 단서의 해석 깊이가 과하게 앞서가지 않는가?
- Phase 0/1/2 결과 보고서 표시 차이가 실제 화면 형태로 작성되었는가?
- 스탯 부족/초과를 태그로 쓰지 않았는가?
- `Visibility`를 장비 속성으로 처리했는가?
- 이력서, 미션 카드, 매칭 화면의 역할을 침범하지 않았는가?
- 결과가 용병 이력서 현재 상태나 장비 상태에 반영되는가?
- 결과 로그를 읽고 플레이어가 다음 준비를 추론할 수 있는가?
- 판정표가 결과 로그를 대체하고 있지 않은가?
- 기존 상태와 이번 변화가 분리되어 표시되는가?
- 플레이어용 단서와 내부 시스템 플래그가 분리되어 있는가?

## 14. 현재 단계에서 하지 않는 것

- 최종 성공률 산식 확정.
- 모든 결과 로그 문체 확정.
- 실제 결과 보고서 대량 작성.
- 모든 부상/파손/상태 목록 확정.
- 경제 밸런스 확정.

지금은 파견 후 구조를 검증할 수 있는 결과 보고서 규격을 잡는 단계다.
