# 태그 시스템 기본 구조

이 문서는 태그를 실제로 만들기 전에 공유해야 하는 기준 문서다.

현재 태그 사전의 목적은 태그 목록을 많이 만드는 것이 아니다. 먼저 **태그가 무엇을 판단하게 만들고, 어떤 문서와 연결되는지**를 정한다.

## 1. 핵심 정의

이 프로젝트에서 태그는 단순한 장식 라벨이 아니다.

태그는 픽서가 용병 이력서, 장비, 미션 카드, 지역, 팩션 정보를 읽고 판단하게 만드는 짧은 정보 단위다.

태그의 역할:

- 용병과 미션의 적합성 판단.
- 장비와 임플란트의 조건/시너지/리스크 판단.
- 지역과 팩션의 위험, 출신, 평판 압축.
- 결과 보고서에서 성공/실패 이유 설명.
- 캐릭터 이야기와 장기 이벤트의 단서 제공.

## 2. 가장 중요한 원칙

태그 자체는 단순해야 한다.

복잡한 해석은 태그 안에 넣지 않고, 다음 문서와 시스템이 담당한다.

```text
단순한 태그
→ 미션별 해석 규칙
→ 이야기 진행 규칙
→ 정보 노출 규칙
```

즉 태그 하나가 모든 효과를 직접 품으면 안 된다.

예:

```text
나쁜 방향:
태그 하나가 전투 보너스, 잠입 페널티, 서사 분기, 경제 비용을 전부 직접 가진다.

좋은 방향:
태그는 짧은 속성으로 두고, 전투 미션/잠입 미션/상층 사교 미션이 각자 다르게 읽는다.
```

## 3. 태그 적용 구조

태그는 단어 하나만 보지 않는다.

항상 아래 구조로 본다.

```text
태그 적용 = 대상 + 관계 + 해석
```

- 대상: 용병, 장비, 임플란트, 미션, 지역, 팩션, NPC, 상태.
- 관계: 출신, 보유, 요구, 위험, 대응, 평판, 소속, 노출, 계약.
- 해석: 지금 미션/스토리/정보 단계에서 어떻게 읽히는가.

같은 단어라도 대상과 관계가 달라지면 의미가 달라진다.

따라서 특정 지역/팩션 고유 구조를 무리하게 공용 태그로 올리면 안 된다.

## 4. 시스템 태그와 이야기 태그

태그는 크게 두 축으로 나눈다.

### 시스템 태그

게임 판정에 직접 쓰이는 태그다.

주요 사용처:

- 미션 진입 조건.
- 기본 관문 성공률 보정.
- 돌발 노드 발생/완화.
- 장비/임플란트 시너지.
- 비용, 보험, 유지비, 보상.
- 신규 상태와 후유증.

시스템 태그는 플레이어 판단에 필요한 정보다. 단, 공개 시점은 `정보 노출 강도 규칙.md`를 따른다.

### 이야기 태그

캐릭터 서사와 장기 이벤트에 쓰이는 태그다.

주요 사용처:

- 과거, 비밀, 관계.
- 특정 NPC/팩션과의 반응.
- 장기 이벤트 플래그.
- 배신, 복수, 구원, 회수, 폭로 같은 이야기 흐름.

이야기 태그는 숫자 최적화 대상으로 전부 공개하면 안 된다. 일부는 Phase 2에서도 숨길 수 있다.

## 5. 스탯과 태그의 역할 분리

5대 스탯은 용병의 기본 성능이다.

- 프레임: 육체, 하드웨어 수용성, 생존, 중화기.
- 쿨: 침착함, 정신 마모 저항, 협상, 사이버싸이코 억제.
- 와이어: 네트워크, 해킹, 기계 조작.
- 사이퍼: 은닉, 흔적 제거, 현장 분석, 잠입.
- 펄스: 직관, 돌발 감지, 비정형 위기 대응.

태그는 스탯을 대체하지 않는다.

태그는 다음 일을 한다.

- 스탯이 어떤 상황에서 의미 있는지 바꾼다.
- 낮은 스탯 용병에게 특정 미션의 조커 카드 역할을 준다.
- 높은 스탯 용병에게도 망설일 위험을 만든다.
- 결과 보고서에 "왜 그렇게 됐는지"를 설명한다.

## 6. 미션이 태그를 읽는 방식

미션은 태그를 고정 효과로 읽지 않는다.

미션은 자기 목적과 노드 구조에 맞춰 태그를 해석한다.

해석 방향:

- 긍정: 이 미션에서 유리하게 작동.
- 부정: 이 미션에서 위험으로 작동.
- 무시: 이 미션에서는 중요하지 않음.
- 특수: 특정 노드나 이야기 조건에서만 반응.

예:

```text
전투 미션:
외형 정보보다 화력, 생존, 장비 운용, 전면 교전 적성이 중요할 수 있다.

잠입/스파이 미션:
외형, 가시성, 의전, 검문, 신분, 말투 같은 정보가 중요해질 수 있다.
```

## 7. 미션 노드와 태그

미션은 단일 판정이 아니라 여러 노드로 진행된다.

기본 흐름:

```text
미션 타입 확인
→ 주요 스탯과 장비 조건 확인
→ 기본 관문 생성
→ 관문 돌파
→ 실패/조건 부족 시 돌발 노드
→ 조커 카드 개입
→ 결과 보고서와 상태 변화
```

태그는 여기서 세 위치에 개입한다.

- 진입 조건: 출격 가능/불가, 경고, 준비 필요.
- 관문 해석: 스탯과 장비의 유리/불리 판정.
- 위기 대응: 실패를 무마하는 조커 카드나 추가 리스크.

## 8. 정보 노출과 태그

태그가 존재한다고 항상 보여주는 것은 아니다.

공개 기준은 `정보 노출 강도 규칙.md`를 따른다.

- Phase 0: 직접 태그명 금지. 텍스트로 암시.
- Phase 1: 정답 대신 신호.
- Phase 2: 시스템 요구와 충돌 이유 공개.
- 신규 발생 상태: 어떤 Phase에서도 즉시 공개.
- 이야기 태그: 일부는 끝까지 숨길 수 있음.

화면 표시량은 `이력서 및 미션 카드 압축 규칙.md`를 따른다.

## 9. 공용 태그와 전용 태그

공용 태그는 "모든 곳에 막 붙는 태그"가 아니다.

공용 태그는 여러 대상이 같은 판단 질문을 공유할 때만 인정한다.

전용 태그는 특정 대상, 지역, 팩션, 장비, 인물군에서만 의미가 선명할 때 사용한다.

판정 기준:

```text
여러 대분류에서 같은 판단 질문을 만들면 공용 후보.
특정 지역/팩션/인물 구조에 묶이면 전용 후보.
같은 단어라도 대상마다 뜻이 갈라지면 관계를 분리.
```

상층 인물 성향 좌표 같은 구조는 중요한 자료지만, 그대로 공용 태그 목록이 되면 안 된다. 상층/인물/팩션 전용 규칙의 재료로 둔다.

## 10. 현재 우선순위

태그 사전 작성 순서는 다음이 안전하다.

1. 태그 시스템 기본 구조 확정.
2. 태그 작성 양식 확정.
3. 태그 관계도 확정.
4. 시스템 태그 규칙 작성.
5. 이야기 태그 규칙 작성.
6. 미션 태그 해석 규칙 작성.
7. 태그 분류 규칙 작성.
8. 미션 카드 제작 규격 작성.
9. 용병 이력서 제작 규격 작성.
10. 결과 보고서 제작 규격 작성.
11. 테스트 미션/이력서/결과 보고서 작성.
12. 실제 태그 목록은 테스트 미션/이력서/결과 보고서 검증 뒤 작성.

## 11. 금지 사항

- 실제 태그를 대량 생성하지 않는다.
- 태그 하나에 복잡한 효과를 모두 넣지 않는다.
- 지역/팩션 전용 구조를 공용 태그로 무리하게 올리지 않는다.
- Phase 0에서 정답 태그명을 직접 보여주는 식으로 설계하지 않는다.
- 시스템 태그와 이야기 태그를 한 문서 안에서 구분 없이 섞지 않는다.
