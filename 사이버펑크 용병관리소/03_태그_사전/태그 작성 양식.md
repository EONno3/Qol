# 태그 작성 양식

이 문서는 실제 태그 목록을 작성하기 위한 문서가 아니다.

현재 단계의 목적은 **태그를 만들기 전, 공용/전용 태그의 생성 규칙과 판정 기준을 고정하는 것**이다.

## 1. 작성 순서

태그 사전은 아래 순서로 작성한다.

```text
태그 시스템 기본 구조
→ 태그 작성 양식
→ 태그 관계도
→ 시스템 태그 규칙
→ 이야기 태그 규칙
→ 미션 태그 해석 규칙
→ 태그 분류 규칙
→ 미션 카드 제작 규격
→ 용병 이력서 제작 규격
→ 결과 보고서 제작 규격
→ 테스트 미션/이력서
→ 실제 태그 목록
```

실제 태그 목록은 테스트 미션/이력서/결과 보고서 검증 뒤에 작성한다.

## 2. 공용 태그 규칙 양식

공용 태그 규칙 문서는 태그 이름을 많이 적는 문서가 아니다.

어떤 종류의 태그를 공용으로 인정할지 정하는 문서다.

```text
## 관계 묶음명

### 사용 대상
- 용병
- 미션
- 지역
- 장비
- 임플란트
- 팩션
- NPC
- 상태

### 핵심 판단 질문
이 관계 묶음이 플레이어에게 어떤 판단을 시키는가?

### 공용으로 인정하는 조건
- 최소 2개 이상의 대분류에서 같은 판단 질문이 반복된다.
- 대상이 달라도 핵심 의미가 유지된다.
- 특정 지역/팩션/인물 고유 설정에 묶이지 않는다.
- 미션 해석 규칙에서 여러 번 재사용 가능하다.

### 공용으로 두면 안 되는 경우
- 특정 계층, 팩션, 지역의 고유 문화일 때.
- 같은 단어지만 대상마다 뜻이 크게 달라질 때.
- 세계관 설명은 풍부하지만 시스템 판단에 거의 쓰이지 않을 때.
- 실제로는 전용 태그 규칙의 재료일 뿐일 때.

### 이름 작성 원칙
- 짧고 직관적으로 작성한다.
- 대상 관계가 불명확하면 관계명을 붙여 분리한다.
- 지역명/팩션명/고유명사는 되도록 전용 태그로 보낸다.

### 전용 규칙으로 내려보낼 재료
- 특정 지역 구조.
- 특정 팩션 문화.
- 특정 계층 성향 좌표.
- 특정 장비 제조사 고유 기믹.
```

## 3. 전용 태그 규칙 양식

전용 태그 규칙 문서는 특정 대상에서만 의미가 선명한 태그를 만들기 위한 기준 문서다.

```text
## 전용 대상

예:
- 용병
- NPC
- 팩션
- 지역
- 미션
- 무기
- 임플란트
- 상태

### 이 대상에서 태그가 필요한 이유
이 대상의 어떤 판단을 압축하기 위해 태그를 쓰는가?

### 참조할 공용 관계
어떤 공용 관계 묶음에서 재료를 가져오는가?

### 전용으로 인정하는 조건
- 해당 대상에서만 의미가 선명하다.
- 공용 태그로 올리면 의미가 흐려진다.
- 특정 지역/팩션/장비/인물 구조와 강하게 묶인다.
- 플레이어가 이 대상의 카드/이력서/브리핑에서 판단해야 한다.

### 금지 사항
- 공용 태그와 같은 뜻을 이름만 바꿔 중복 생성하지 않는다.
- 설정 설명을 태그로 과하게 쪼개지 않는다.
- 시스템 태그와 이야기 태그를 한 항목에 섞지 않는다.

### 실제 태그 작성 전 검토 질문
- 이 태그는 어디에 붙는가?
- 플레이어가 무엇을 판단하게 만드는가?
- 어느 미션/노드/이야기에서 읽히는가?
- Phase 0/1/2 중 언제 공개되는가?
```

## 4. 시스템 태그 판정 양식

시스템 태그는 게임 판정에 직접 영향을 주는 태그다.

```text
## 시스템 태그 후보 판정

### 붙는 대상

### 읽는 시스템
- 미션 진입 조건
- 기본 관문
- 돌발 노드
- 조커 카드
- 장비 시너지
- 비용/보험/유지비
- 결과 보고서 상태

### 판단 효과
이 태그가 플레이어에게 어떤 선택을 하게 만드는가?

### 공개 단계
- Phase 0: 암시 가능 여부
- Phase 1: 신호 표시 여부
- Phase 2: 명확 공개 여부

### 주의
수치 효과를 문서에 먼저 박지 않는다.
정확한 수치는 밸런싱 단계에서 따로 정한다.
스탯값 자체를 태그로 만들지 않는다.
예: `#낮은_Frame`, `#높은_Cypher`, `#Wire_경계값` 금지.
스탯 부족/초과는 태그가 아니라 수치/등급/매칭 판정으로 표시한다.
```

## 5. 이야기 태그 판정 양식

이야기 태그는 서사, 관계, 비밀, 장기 이벤트에 쓰이는 태그다.

```text
## 이야기 태그 후보 판정

### 붙는 대상

### 이야기 기능
- 과거
- 비밀
- 관계
- 복수/배신/구원
- 특정 팩션 반응
- 장기 이벤트

### 시스템과의 접점
시스템 판정에 직접 쓰이는가, 아니면 결과 보고서/후속 이벤트에만 쓰이는가?

### 공개 단계
- 이력서 암시
- 특정 미션 반응
- 결과 보고서 단서
- 후속 이벤트 공개

### 주의
이야기 태그는 분석기만으로 전부 공개하지 않는다.
Phase 2에서도 일부 핵심 비밀은 숨길 수 있다.
```

## 6. 실제 태그 카드 양식

실제 태그 목록을 만들 단계가 오면 아래 축약 양식을 쓴다.

```text
## 태그명

- 구분: 시스템 / 이야기 / 혼합 후보
- 분류: 공용 / 전용
- 적용 대상:
- 핵심 의미:
- 읽는 시스템 또는 이야기:
- 공개 단계:
- 사용 금지/주의:
```

혼합 후보는 바로 확정하지 않는다.

시스템 태그와 이야기 태그로 나눌 수 있는지 먼저 검토한다.

## 7. 공용/전용 빠른 판정

```text
Q1. 여러 대상에서 같은 판단 질문을 만드는가?
→ 예: 공용 후보.

Q2. 특정 지역/팩션/인물/장비에 묶여야만 의미가 사는가?
→ 예: 전용 후보.

Q3. 미션 성공률, 출격 조건, 비용, 상태에 직접 쓰이는가?
→ 예: 시스템 태그 후보.

Q4. 관계, 비밀, 과거, 장기 이벤트에 쓰이는가?
→ 예: 이야기 태그 후보.

Q5. 지금 당장 실제 태그명을 많이 만들어야 하는가?
→ 아니오. 먼저 규칙과 테스트 구조를 만든다.
```

## 8. 현재 단계에서 하지 않는 것

- 실제 공용 태그 대량 작성.
- 용병 전용 태그 대량 작성.
- 상층/중층/하층 고유 구조를 공용 태그로 직접 승격.
- 미션 카드 문체 확정.
- 이력서 말투 확정.
- 수치 밸런스 확정.
