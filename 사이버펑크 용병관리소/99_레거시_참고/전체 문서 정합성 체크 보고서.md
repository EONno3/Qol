## 3. 문서 간 공통 전제

다음 전제는 여러 문서에서 반복 확인된다.

- 게임은 텍스트 기반 부분 방치형 용병 파견 매니지먼트다.
- 플레이어는 픽서다.
- 핵심 루프는 미션 확인, 용병 판독, 장비/스테이션 준비, 파견, 노드 판정, 결과 보고서, 보상/상태 변화다.
- 용병은 5대 스탯과 태그, 장비, 임플란트, 상태, 이야기 단서를 가진다.
- 미션은 전투, 잠입, 지원, 기업, 비밀 5종이다.
- 미션은 단일 판정이 아니라 기본 관문, 돌발 노드, 조커 카드로 진행된다.
- 태그는 단순하고, 미션/이야기가 복잡하게 해석한다.
- 정보는 Phase 0/1/2로 점진 공개된다.
- 도시는 하층, 중층, 상층, 정부 기관으로 나뉜 돔 도시다.
- 팩션은 메가코프, CEO 가문, 상층 권력층, 중층 독립 팩션, 하층 인프라 팩션으로 정리된다.

## 4. 충돌 및 정리 필요 항목

### 4.1 Phase 용어 충돌

이전 상태에서 `02_핵심_시스템/픽서.md`는 미션 판정 과정을 Phase 1~4로 불렀다.

태그 사전과 예시 문서는 정보 해금 단계를 Phase 0~2로 부른다.

충돌 위험:

- AI가 `Phase 1`을 정보 해금인지 미션 판정인지 혼동할 수 있다.

정리 기준:

- `Phase 0~2`는 정보 해금 전용 용어.
- 미션 쪽은 `판정 단계 1~4` 또는 `미션 처리 단계`로 부른다.
- 현재 `02_핵심_시스템/픽서.md` 상단에 이 기준을 명시했고, 내부 미션 진행 용어도 `판정 단계 1~4`로 정리했다.

### 4.2 구형 태그 철학과 최신 태그 철학

`99_레거시_참고/기획 회의 사항.md`, `02_핵심_시스템/용병.md`, `99_레거시_참고/태그 및 스탯 시스템 구조안.md`는 태그를 긍정/부정/이벤트/스탯 특화 중심으로 적는다.

최신 태그 사전은 시스템 태그/이야기 태그 분리, 단순 태그, 미션별 해석을 기준으로 한다.

정리 기준:

- 구형 문서는 아이디어 저장소와 초기 분류 자료로 본다.
- 최신 기준은 현재 구조의 `03_태그_사전/` 문서군이다.
- 현재 `99_레거시_참고/기획 회의 사항.md`와 `99_레거시_참고/태그 및 스탯 시스템 구조안.md` 상단에 레거시/초기 자료 역할을 명시했다.

### 4.3 `02_핵심_시스템/픽서.md` 파일명과 내용 불일치

`02_핵심_시스템/픽서.md`의 실제 내용은 픽서 개인 설정이 아니라 미션 타입과 미션 판정 로직이다.

정리 기준:

- 온톨로지에서는 `02_핵심_시스템/픽서.md`를 `미션/노드 판정 원본`으로 참조한다.
- 현재 `02_핵심_시스템/픽서.md` 상단에 문서 역할을 명시했다.
- 추후 파일명을 `미션.md` 또는 `미션 판정.md`로 변경할지 검토한다.

### 4.4 도시/하층 인구 수 차이

`01_세계관/도시/AI가 이해한 도시 구성.md`는 하층 인구 약 600만 명.

`01_세계관/팩션/하층.md`는 UNzone 인구 약 350만 명.

`01_세계관/도시/도시 구상도.md` 원본 텍스트에는 하층 600만 명, UNzone 400만 명, Under-Lid 200만 명이 나온다.

정리 기준:

- 하층 전체: 약 600만 명.
- UNzone: 약 350~400만 명 후보.
- Under-Lid/하층 인프라 구역: 약 200만 명 후보.
- 정확한 수치는 추후 확정 필요.

### 4.5 UI 목업과 최신 정보 해금 규칙 차이

`02_핵심_시스템/사펑용병.html`은 초기 UI 목업이다.

현재 목업은 미션 리스트, 용병 리스트, 보수 분배, 만족도, 복지 서비스를 보여준다. Phase 0/1/2, 태그 신호, 위험 노드 UI는 아직 반영되지 않았다.

정리 기준:

- UI 목업은 게임 루프와 테마 참고.
- 최신 판독 UI 기준은 태그 사전의 정보 노출/압축 규칙.

### 4.6 실제 태그 예시와 규칙 문서의 관계

`99_레거시_참고/미션 & 용병 예시.md`에는 `#마사다_출신`, `#고가시성_의수`, `#호흡계_마모` 같은 예시 태그가 나온다.

정리 기준:

- 예시 태그는 더미다.
- 실제 태그 생성 규칙이 아니다.
- Phase 흐름과 UI 노출 테스트용으로만 본다.

### 4.7 테스트 샘플에서 확정된 정보 위치 규칙

2차/3차 테스트 샘플을 통해 아래 원칙이 확정 후보가 되었다.

- 용병 이력서는 상시 고용 정보다. 특정 미션에 맞춘 해석을 쓰지 않는다.
- 미션 카드는 의뢰 정보다. 특정 용병을 전제로 쓰지 않는다.
- 용병과 미션의 연결 해석은 미션 슬롯 매칭 화면에서만 보여준다.
- 스탯은 태그가 아니다. 스탯 부족/초과는 수치, 등급, 매칭 판정으로 처리한다.
- `Visibility`는 긍정/부정 태그가 아니라 장비 속성/수치 후보다.
- `보유 태그 상세 내역`에는 스탯 한계나 스탯 수치를 다시 쓰지 않는다.
- Phase 1은 성공 전망과 대표 신호, Phase 2는 정확한 조건 충돌과 예상 성공률을 보여준다.

## 6. 온톨로지 갱신 기준

온톨로지는 아래를 반영해야 한다.

- 태그 사전만이 아니라 미션 카드/용병 이력서/결과 보고서 제작 규격, 테스트 샘플과 비교 검토까지 포함한 전체 40개 문서를 기준으로 한다.
- `03_태그_사전/`은 태그 철학과 정보 해금의 최신 기준이다.
- 2차/3차 테스트 샘플과 비교 검토 문서는 미션 카드, 용병 이력서, 미션 슬롯 매칭의 정보 위치 검증 자료다.
- 1차 테스트 샘플은 초안/오류 발견 자료이며 최신 기준으로 삼지 않는다.
- `02_핵심_시스템/픽서.md`는 이름과 달리 미션/노드 판정 원본이다.
- `99_레거시_참고/태그 및 스탯 시스템 구조안.md`와 `99_레거시_참고/기획 회의 사항.md`는 초기 아이디어/레거시 문서다.
- `02_핵심_시스템/사펑용병.html`은 UI 테마와 기본 루프 목업이다.
- 도시/팩션 문서는 세계관과 지역/팩션 전용 태그의 원천이다.

## 7. 정합성 기준 다음 작업

1. `테스트 결과 보고서 1차 샘플.md`를 검토해 결과 보고서 규격의 부족 항목을 확인한다.
2. 필요하면 `결과 보고서 제작 규격.md`를 보강한다.

4. 결과 보고서까지 이어지는 흐름을 확인한 뒤 실제 태그 목록/용병 풀/미션 풀 작성으로 넘어간다.
5. 별도 정리 이슈로 `02_핵심_시스템/픽서.md` 파일명 변경 여부와 하층 인구 수 확정이 남아 있다.
