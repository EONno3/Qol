# 2차-3차 샘플 비교 검토

이 문서는 `테스트 미션 및 용병 2차 샘플.md`와 `테스트 미션 및 용병 3차 샘플.md`를 비교해, 다음 제작 단계로 넘길 규칙을 정리한다.

중요:

- 두 샘플의 태그명, 수치, 성공 전망, 성공률은 더미다.
- 목적은 콘텐츠 확정이 아니라 정보 위치와 제작 규격 검증이다.
- 1차 샘플은 초안/오류 발견 자료로만 본다.

## 1. 샘플 역할

### 2차 샘플

- 미션: `7번 바이패스 밸브 차단`
- 검증 축: 환경 위험, 정비, 검문, `Frame`, `Wire`, `Visibility`, 호흡 대응
- 핵심 질문: 물리 작업/기술 작업/장비 노출/환경 대응이 한 미션 안에서 분리되어 읽히는가?

### 3차 샘플

- 미션: `실버램프 야간 경비 대행`
- 검증 축: 경비, 오인식별, 제한 교전, `Cypher`, `Cool`, `Pulse`, `Visibility`
- 핵심 질문: 평범한 경비 미션처럼 보이는 의뢰에서 비전투 스탯의 필요성이 개연성 있게 드러나는가?

## 2. 공통으로 작동한 구조

### 2.1 이력서

작동한 점:

- 이력서는 상시 고용 정보로 분리되었다.
- 특정 미션 목표나 장치를 직접 해결하는 문장을 제거했다.
- `스탯 프로필`과 `보유 태그 상세 내역`이 분리되었다.
- 스탯 수치/스탯 한계는 보유 태그 상세에 들어가지 않는다.
- Phase 0은 사건/습관/장비 단서, Phase 1은 대표 신호, Phase 2는 상세 태그와 확장 이력 구조로 작동한다.

유지할 규칙:

- 이력서에는 미션 적합/부적합 해석을 쓰지 않는다.
- 이력서에는 특정 미션의 목표, 장소, 장치, 위험, 보상을 직접 언급하지 않는다.
- `보유 태그 상세 내역`은 태그, 장비 속성, 상태, 이야기/태생 정보만 담는다.
- 스탯 수치는 `스탯 프로필`에만 둔다.

### 2.2 미션 카드

작동한 점:

- 미션마다 요구 스탯/장비/속성의 개연성을 미션 내부 사정으로 설명했다.
- Phase 0 브리핑은 조건명을 직접 쓰지 않고, 상황과 흔적으로 요구 방향을 암시했다.
- Phase 2는 필수/권장/위험/추가 베네핏을 명확히 보여줬다.
- 미션 카드가 특정 용병을 전제로 쓰이지 않았다.

유지할 규칙:

- 스탯이나 속성을 요구하려면 왜 필요한지 미션 내부 원인을 둔다.
- 미션 카드는 의뢰 자체의 정보만 담는다.
- 특정 용병과 연결한 해석은 미션 카드 본문에 쓰지 않는다.

### 2.3 미션 슬롯 매칭

작동한 점:

- 용병과 미션의 연결 해석이 슬롯 대입 화면으로 분리되었다.
- Phase 1은 성공 전망과 흐릿한 신호만 보여준다.
- Phase 2는 정확한 조건 충돌, 충족 여부, 예상 성공률을 보여준다.
- 스탯 부족/초과 해석은 매칭 화면에서만 처리된다.

유지할 규칙:

- 매칭 화면은 `미션 카드 + 용병 이력서`를 연결하는 해석 레이어다.
- 캐릭터 창이나 미션 카드 본문에 매칭 결론을 넣지 않는다.
- Phase 1 성공 전망은 범위/등급으로 둔다.
- Phase 2 예상 성공률은 더미 단계에서는 숫자 감각만 테스트한다.

## 3. 확정 후보 규칙

### 3.1 태그/스탯/속성 경계

- 스탯은 태그가 아니다.
- `#낮은_Frame`, `#높은_Cypher`, `#Wire_경계값` 같은 태그는 만들지 않는다.
- 스탯 부족/초과는 매칭 화면에서 `Frame 36 → 요구 60 미충족`처럼 판정한다.
- `Visibility`는 장비 속성/수치 후보다.
- `Visibility`가 높은 장비는 항상 부정이 아니다. 검문/잠입/기업/보안 미션에서 위험값으로 읽힌다.
- 태그는 출신, 경험, 성향, 장비 기능, 상태, 이야기/태생 정보처럼 스탯값으로 환원되지 않는 정보에 붙인다.

### 3.2 정보 위치

- 용병 이력서: 상시 고용 정보.
- 미션 카드: 의뢰 정보.
- 미션 슬롯 매칭: 특정 미션과 특정 용병의 연결 해석.
- 결과 보고서: 파견 후 실제 발생한 결과와 정산.

### 3.3 Phase 정보 노출

- Phase 0: 텍스트 암시.
- Phase 1: 대표 신호, 미확인 위험, 성공 전망.
- Phase 2: 상세 조건, 태그/속성/상태 공개, 정확한 충돌 설명.

## 4. 반복된 실수와 방지 규칙

### 4.1 이력서에 미션 해석 삽입

문제:

- 이력서가 특정 미션을 위해 작성된 것처럼 보였다.
- 용병 설명에 미션 목표/장치/위험을 직접 해결하는 문장이 들어갔다.

방지 규칙:

- 이력서는 미션 목록을 보기 전에도 자연스럽게 읽혀야 한다.
- 이력서의 모든 문장은 상시 고용 정보로 설명 가능해야 한다.
- 미션 요구와 용병의 연결은 매칭 화면으로 보낸다.

### 4.2 스탯값 태그화

문제:

- 낮은 스탯을 `#낮은_Frame` 같은 태그로 만들었다.
- 보유 태그 상세에 스탯 한계를 다시 적었다.

방지 규칙:

- 스탯 수치와 등급은 스탯 프로필에만 둔다.
- 스탯 한계는 매칭 화면에서만 해석한다.
- 보유 태그 상세에는 스탯 한계 항목을 만들지 않는다.

### 4.3 Visibility 태그화

문제:

- 가시성을 부정 태그처럼 다뤘다.

방지 규칙:

- `Visibility`는 장비 속성/수치 후보다.
- 가시성이 부정인지 아닌지는 미션이 해석한다.

### 4.4 캐릭터 창에 매칭표 삽입

문제:

- 캐릭터 Phase 2에 잘 맞는 미션/위험한 미션/조커 카드 가능성 같은 항목이 들어갔다.

방지 규칙:

- 캐릭터 창에는 캐릭터 정보만 둔다.
- 미션 적합성은 미션 슬롯 매칭으로만 표시한다.

## 5. 아직 미정인 것

- Phase 1 기호 `🟢/🔴/△/▽/❓`의 정확한 의미.
- 성공 전망 등급과 Phase 2 예상 성공률의 산식.
- 결과 보고서가 어떤 순서와 문체로 성공/실패를 설명할지.
- 신규 상태, 부상, 파손, 평판 변화의 정산 형식.
- 보험/수리/치료/휴식 비용이 결과 보고서에서 어떻게 표시될지.
- 이야기 태그가 결과 보고서와 후속 이벤트에서 어느 정도까지 드러날지.

## 6. 다음 단계 판단

지금 바로 4차 샘플을 만드는 것은 우선순위가 낮다.

이유:

- 파견 전 정보 구조는 2차/3차에서 어느 정도 검증되었다.
- 2차 미션 기준 결과 보고서 1차 샘플이 생겼다. 다른 미션 유형 검증은 아직 남아 있다.
- 게임의 핵심 루프는 결과 보고서와 정산까지 이어져야 검증된다.

다음 단계:

1. `테스트 결과 보고서 1차 샘플.md`를 검토한다.
2. 결과 보고서 규격에 부족한 항목이 있으면 보강한다.
3. 3차 미션 기준 결과 보고서 샘플을 추가해 경비/오인식별/제한교전형에서도 작동하는지 확인한다.
4. 결과 보고서 샘플 검증 뒤 관련 규격과 온톨로지를 다시 갱신한다.

## 7. 현재 결론

2차/3차 샘플은 파견 전 정보 구조 검증용 기준 후보로 유지한다.

다만 최종 콘텐츠 예시는 아니다.

다음 검증 대상은 `테스트 결과 보고서 1차 샘플`의 판독성과 규격 적합성이다.
