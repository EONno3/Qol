## 1. 총평

현재 MVP 문서군은 **기획 검증용 MVP로는 통과 가능한 상태**다.

미션 선택, 용병 확인, 분석 단계, 매칭, 출격 판정, 결과 보고서, 다음 반영까지 한 번의 루프가 문서상으로 닫힌다.

다만 아직 **실제 구현용 데이터 규격**은 아니다. 지금 문서들은 플레이 테스트 기준과 사례집에 가깝고, 코드나 데이터 테이블로 옮기려면 필드명, 값 범위, 상태 변화 규칙을 더 단단하게 잘라야 한다.

---

## 2. 잘된 점

### 2.1 루프가 닫혔다

현재 문서군은 아래 흐름을 모두 가진다.

```text
미션 카드
-> 용병 이력서
-> 매칭 케이스
-> 출격 판정
-> 결과 보고서
-> 상태/장비/평판/후속 단서 반영
```

이전처럼 설정만 쌓이는 상태가 아니라, 플레이어가 하나의 선택을 하고 그 선택의 결과를 받을 수 있는 구조가 생겼다.

### 2.2 MVP 범위가 비교적 잘 지켜졌다

`MVP_테스트_세트.md`는 하층/중층/상층 각 1개 대표 미션만 사용한다.

용병도 4명으로 제한되어 있다.

- 하층 적합형
- 중층 적합형
- 상층 적합형
- 실패 유도형

이 구성은 최소 검증 세트로 적절하다. 더 늘리면 지금 단계에서는 밸런스 작업처럼 번질 가능성이 크다.

### 2.3 이력서와 매칭 화면의 역할이 분리됐다

`MVP_테스트_용병_이력서.md`는 용병을 특정 미션의 정답표처럼 쓰지 않는다.

반대로 `MVP_매칭_케이스표.md`는 실제 대입 시 어떤 신호가 나오는지 분리해서 다룬다.

이 분리는 중요하다. 이 구조가 유지되면 플레이어는 이력서를 읽고 가설을 세운 뒤, 매칭 화면에서 그 가설을 확인하는 흐름을 경험할 수 있다.

### 2.4 실패 케이스가 포함됐다

`크롬쇼`는 단순한 약한 용병이 아니라, 특정 상황에서는 쓸 수 있지만 침투/위장 미션에는 위험한 테스트 말이다.

이 덕분에 MVP는 성공만 검증하지 않고 아래를 같이 확인할 수 있다.

- 출격 불가
- 치명적 경고
- 강제 출격 시 실패
- 장비 손상
- 평판 손실
- 상태 태그 악화

### 2.5 결과 보고서가 다음 선택으로 돌아간다

`MVP_결과_보고서_샘플.md`는 보상만 적지 않고, 상태/장비/평판/후속 단서를 다음 반영으로 돌려보낸다.

이 점은 게임 루프 관점에서 강점이다. 결과 보고서가 단순 엔딩 로그가 아니라 다음 턴의 판단 재료가 된다.

## 3. 부족한 점

### 3.1 아직 판정 수식이 구현 가능한 형태는 아니다

현재 매칭 케이스는 기획 판단으로는 충분하지만, 구현 데이터로는 부족하다.

예:

- `성공 우세`
- `부분 성공 위험`
- `손익 비효율`
- `고위험 조건부 출격`

이 표현들은 사람이 읽기에는 좋지만, 시스템 판정값으로 쓰기에는 애매하다.

MVP 구현 전에는 최소한 아래 정도의 정규값이 필요하다.

```text
출격 판정: 가능 / 경고 / 불가
성공 전망: 높음 / 보통 / 낮음
손실 전망: 낮음 / 보통 / 높음
결과 타입: 성공 / 부분 성공 / 실패 / 조기 철수 / 사고
```

### 3.2 장비 보유 상태가 완전히 구조화되지 않았다

문서에는 절연 안전화, 은닉형 슬롯, 상층 기압 보정기, DNA 패치가 등장한다.

하지만 MVP 데이터로는 아직 아래가 확정되지 않았다.

- 장비가 용병 기본 보유인지
- 미션 전 대여/장착 가능한지
- 장착 슬롯 제한이 있는지
- 장비 파손이 다음 미션에 어떻게 영향을 주는지

이 부분은 다음 단계에서 최소 장비 목록으로 정리해야 한다.

### 3.3 상태 태그의 효과가 아직 느슨하다

결과 보고서에는 `#수면부족_심화`, `#탄환_관통흔`, `#전뇌_단락_충격` 같은 상태가 나온다.

하지만 각 상태가 다음 미션에서 어떤 식으로 적용되는지는 아직 예시 수준이다.

MVP에서는 모든 상태를 대량 사전화할 필요는 없지만, 테스트에 쓰인 상태만큼은 최소 효과가 필요하다.

### 3.4 결과 보고서 Phase 구조는 아직 MVP 샘플에 약하게만 반영됐다

결과 보고서 제작 규격에는 결과 보고서도 Phase별 표시 구조를 가진다고 되어 있다.

현재 MVP 결과 샘플은 플레이어가 보기 좋은 전체 보고서 형태에 가깝고, 결과 보고서 Phase 0/1/2 분리는 아직 명확하지 않다.

다만 MVP 첫 검증에서는 이 정도가 허용된다. 먼저 결과 루프가 닫히는지 보는 것이 더 중요하다.

### 3.5 경제 밸런스는 아직 더미다

보상, 정비비, 손실 수치는 플레이 감각을 보기 위한 임시값이다.

현재 단계에서 경제 밸런스를 확정하려 하면 MVP 범위를 벗어난다. 다만 “손실이 있다는 느낌”과 “성공해도 비용이 남는 구조”는 잘 드러난다.

```powershell

# 브라우저 http://localhost:5173/
```

확인할 3케이스:

| 조합 | 기대 결과 |
| --- | --- |
| 차단기 × 하층 변전소 | 성공, 정산 후 잔고·상태·평판·후속 단서 반영 |
| 몰트 × 중층 클럽 | 부분 성공 |
| 크롬쇼 × 중층 클럽 | 출격 불가 (슬라이드 잠김) |

판단 질문:

- 미션 → 수주 → 요약 → 매칭 → 출격 → 결과 → 정산 흐름이 자연스러운가?
- 매칭 신호와 결과 보고서가 앞선 선택과 이어지는가?
- `UI/미션 화면 흐름.md`와 체감이 맞는가?

### 5.2 2순위: 플레이테스트 피드백 반영

플레이테스트 후 나온 이슈만 최소 범위로 수정한다.

예상 범위:

- 화면 전환 순서/버튼 위치 조정
- 텍스트 가독성, 정보 노출 타이밍
- 매칭 신호 표현 방식
- (필요 시) seed 데이터 문구/수치 소규모 수정

하지 않을 것 (고도화 후보):

- 2턴 이상 루프, 저장/불러오기, CSV 자동 로더, 캐치업, 상점/정비소 UI

### 5.3 3순위: 다음 방향 결정

플레이테스트 결과에 따라 아래 중 하나를 선택한다.

1. **프로토타입 확장** — 2턴, 후속 단서 미션 연결, 분석 레벨 UI 고도화
2. **실제 게임 클라이언트 이전** — Unity/Godot 등으로 옮기되 MVP 데이터·루프 유지
3. **데이터/기획 수정** — 루프는 맞지만 매칭/결과 문구·구조 재조정

## 7. 결론

MVP 문서·데이터·웹 프로토타입까지 **1턴 루프를 눌러볼 수 있는 상태**다.

다음 단계에서 바로 대량 설정을 만들면 안 된다.  
지금은 **기획자가 웹 프로토타입을 직접 플레이**해 흐름과 가독성을 판단하는 것이 우선이다.

판단이 끝나면 피드백 반영 → 확장/이전/기획 수정 중 하나를 선택한다.
